## 2. 도메인 경계 검증

> Modular Monolith v1의 핵심 방어선이다. v2 MSA 분리 비용을 낮추기 위해 v1부터 도메인 간 직접 의존을 차단한다.

### 2.1 도구 구성

| 도구 | 역할 | 검증 방식 |
| --- | --- | --- |
| Spring Modulith 1.2+ | 메인 — 모듈 경계 + 인프로세스 이벤트 흐름 검증 | `@ApplicationModule` 메타데이터 |
| ArchUnit 1.3+ | 보조 — 금지 import, Controller/Repository 직접 연결, 금지 기술 의존성 검증 | JUnit5 테스트 |

### 2.2 도메인 패키지 메타데이터

각 도메인 패키지의 `package-info.java`에 `@ApplicationModule`을 선언한다.

```java
// com/qtai/domain/bible/package-info.java
@ApplicationModule(displayName = "Bible Domain", allowedDependencies = {})
package com.qtai.domain.bible;

import org.springframework.modulith.ApplicationModule;
```

```java
// com/qtai/domain/qt/package-info.java
@ApplicationModule(
    displayName = "QT Domain",
    allowedDependencies = {
        "bible::api",
        "study::api"
    }
)
package com.qtai.domain.qt;

import org.springframework.modulith.ApplicationModule;
```

| 도메인 | allowedDependencies |
| --- | --- |
| `member` | 없음 — Kakao OAuth/JWK는 `security.oauth` 인프라와 분리 |
| `bible` | 없음 |
| `qt` | `bible::api`, `study::api` |
| `study` | `bible::api`, `ai::api` |
| `note` | `qt::api`, `bible::api` |
| `sharing` | `note::api`, `member::api` |
| `praise` | 없음 또는 `member::api` |
| `ai` | `bible::api`, `qt::api`, `study::api` |
| `admin` | 기능별 도메인의 공개 api만 orchestration |

> `note`, `sharing`, `praise`는 `bible` 하위 모듈이 아니라 별도 도메인이다. 도메인 간 호출은 공개 `api/` Port와 DTO만 사용한다.

### 2.3 공개 인터페이스 = `@NamedInterface("api")`

도메인 간 호출이 허용되는 Port/Facade/DTO는 각 도메인의 `api/` 하위 패키지에 둔다.

```java
// com/qtai/domain/bible/api/package-info.java
@NamedInterface("api")
package com.qtai.domain.bible.api;

import org.springframework.modulith.NamedInterface;
```

다른 도메인은 `com.qtai.domain.bible.api.*`만 import할 수 있다.
`com.qtai.domain.bible.repository.*`, `com.qtai.domain.bible.entity.*`, `com.qtai.domain.bible.service.*` 직접 import는 자동 거절한다.

## 3. 요구사항 정합성 Guard

### 3.1 금지 기술·금지 기능 검사

CI는 아래 패턴을 검사한다. 실제 구현에서는 스크립트, ArchUnit, OpenAPI lint를 조합한다. 경로명은 구현 저장소 구조에 맞게 조정하되, 검사 의도는 유지한다.

```bash
scan_paths=()
for path in qtai-server flutter-app apis infra .github/workflows db data; do
  [ -e "$path" ] && scan_paths+=("$path")
done

[ ${#scan_paths[@]} -eq 0 ] && exit 0

# 사용자용 AI 엔드포인트/SSE 금지
rg -n "/ai/sessions|SseEmitter|text/event-stream|Flux<ServerSentEvent" "${scan_paths[@]}" && exit 1

# RAG/벡터 DB/검색엔진 금지
rg -n "Chroma|chromadb|VectorStore|EmbeddingStore|Elasticsearch|Weaviate|RAG" "${scan_paths[@]}" && exit 1

# Kafka/K8s/Helm v1 금지
rg -n "KafkaTemplate|spring-kafka|kind: Deployment|apiVersion: apps/v1|helm" "${scan_paths[@]}" && exit 1

# 금지 성경 번역본 금지
rg -n "개역개정|ESV|NIV" "${scan_paths[@]}" && exit 1

# 교회 인증 MVP 제외
rg -n "church verification|churchAuth|교회 인증|church_cert" "${scan_paths[@]}" && exit 1

# AI 찬양 추천 금지
rg -n "recommend.*song|song.*recommend|AI.*찬양|찬양.*추천" "${scan_paths[@]}" && exit 1

# 직접 YouTube URL 입력, 가사·음원 저장 금지
rg -n "youtubeUrl|youtube_url|direct.*youtube|lyrics|가사|audioFile|음원" "${scan_paths[@]}" && exit 1
```

`.github/pull_request_template.md`처럼 금지 항목을 체크리스트로 설명하는 문서성 파일은 Requirements Guard 대상에서 제외한다. GitHub 설정 검사는 `.github/workflows`만 대상으로 삼고, PR 템플릿의 문구 정합성은 Document/Terminology Guard와 리뷰로 확인한다.

### 3.2 배치 AI 정책 검사

| 규칙 | 검사 방향 |
| --- | --- |
| 해설·시뮬레이터 요청 경로 생성 금지 | 일반 `@RestController`에서 해설·시뮬레이터 생성용 DeepSeek client 직접 참조 금지 |

### 3.3 시뮬레이터 상태 검사

오늘 QT 응답은 시뮬레이터 클립이 없어도 상태값을 반환해야 한다.

| 상태 | UI |
| --- | --- |
| `READY` | 보기 버튼 활성화 |
| `MISSING` | 보기 버튼 비활성화 |
| `FAILED` | 보기 버튼 비활성화 |
| `DISABLED` | 보기 버튼 비활성화 |

테스트는 `READY/MISSING/FAILED/DISABLED` 모든 케이스를 포함한다.

### 3.4 오늘 QT 캐시·배치 시각 검사

성서 유니온 공개 시각과 우리 시스템 수집 배치 시각은 전 구간에서 분리한다.

| 검사 항목 | 기준 |
| --- | --- |
| 공개 시각 | 00:00 KST |
| 수집 배치 시각 | 04:00 KST |
| 00:00~04:00 응답 | 이전에 준비된 QT 캐시를 반환 |
| 04:00 배치 이후 | 본문 범위 수집, 승인 해설, 시뮬레이터 상태, 캐시 교체 |
| 오늘 QT 100% | 본문, 해설 진입점, 묵상 진입점, 시뮬레이터 상태값이 항상 응답 |

오늘 QT 100%는 모든 본문에 실제 시뮬레이터 클립이 있다는 뜻이 아니다. `READY`가 아닌 상태는 버튼을 비활성화한다.

### 3.5 이벤트 실패 대응 검사

Spring `ApplicationEventPublisher`는 Kafka처럼 영속 메시징을 보장하지 않는다. v1 품질 게이트는 "유실률 0%"가 아니라 "핸들러 실패 로그 + 재처리 가능"을 검사한다.

| 항목 | 기준 |
| --- | --- |
| 실패 로그 | `eventId`, event type, handler name, error message 기록 |
| 재처리 | DB 상태 기준으로 관리자 또는 배치가 재시도 가능 |
| 묵상·알림 이벤트 | `notifications.event_key`, 감사 로그, 도메인 상태 기준으로 추적 가능한 기록 |
| 빈 catch | 금지 |

### 3.6 관리자 권한 검사

| API/작업 | 요구 권한 |
| --- | --- |
| `/api/v1/admin/**` | `ADMIN` |
| AI 배치/수집 배치 | `SYSTEM_BATCH` |
| AI 산출물 저장 | `SYSTEM_BATCH` |
| 오늘 QT/해설/찬양 큐레이션 수정 | `ADMIN` + 감사 로그 |

관리자 작업은 `admin_audit_logs`에 `actorId`, `actorRole`, `action`, `targetType`, `targetId`, `beforeJson`, `afterJson`, `ipAddress`, `userAgent`, `createdAt`을 남긴다.

### 3.7 성경 JSON 소스 등록 검사

성경 본문 데이터는 GitHub 공개 JSON이라도 출처 검토 없이 적재할 수 없다. seed, fixture, 관리자 등록 데이터는 아래 메타데이터를 갖춰야 한다.

| 필드 | 필수 여부 | 기준 |
| --- | --- | --- |
| `repoUrl` | 필수 | GitHub 공개 저장소 URL |
| `license` | 필수 | 라이선스명 또는 검토 상태 |
| `translationName` | 필수 | 번역본명 |
| `attribution` | 필수 | 앱/관리자 화면에 표기할 출처 방식 |
| `redistributionAllowed` | 필수 | 재배포 가능 여부 |
| `prohibitedReason` | 조건부 | 개역개정, ESV, NIV 등 금지 번역본 판정 사유 |

`repoUrl`, `license`, `translationName`, `attribution`, `redistributionAllowed` 중 하나라도 없으면 적재 PR을 머지하지 않는다.

### 9.2 문서 표현 검사

아래 표현은 허용 문맥으로 사용하면 안 된다. `23_도메인_용어사전.md`의 "피해야 할 표현" 표처럼 금지 또는 회피 문맥으로 설명하는 경우만 예외다.

| 검사 대상 | 허용 기준 |
| --- | --- |
| `저작권 문제 없음` | 금지. `저작권 리스크를 낮춘다`로 표현 |
