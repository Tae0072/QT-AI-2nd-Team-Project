## 1. 배경 & 결정

- 강사님 직강 피드백으로 **프론트엔드(Flutter) 정비 5단계**를 우선 진행하기로 한다.
- 이 5단계가 끝나야 이후 **MSA(v2) 분리**가 수월해진다는 판단이다.
- 본 문서는 그 새 지침을 기록한다. 기존 기준 문서와 다를 수 있는 부분은 본 문서를 우선하되, **백엔드 v1 = Modular Monolith 유지** 결정 자체는 바뀌지 않는다.

| 항목 | 기존 규정 | 본 로드맵과의 관계 |
| --- | --- | --- |
| 백엔드 형태 | v1 = 단일 `qtai-server` Modular Monolith · v2 = MSA 분리(2026-06-08) | FE는 게이트웨이 단일 진입 유지 |
| MSA | "v2 분리 비용을 낮추려 v1부터 경계 차단"(18_품질게이트) | 여전히 v2 목표. 본 로드맵은 그 **선행 준비** |
| 금지 기술 | Kafka·Kubernetes·Helm은 v1 제외·v2 MSA 허용(2026-06-08) | — |
| 본 로드맵 범위 | — | **Flutter 앱(프론트엔드) 정비**. 백엔드 v1 규정과 직접 충돌하지 않음 |

> 정리: 이번 작업은 **프론트엔드 코드 정비**다. 백엔드를 지금 MSA로 쪼개는 것이 아니라, 나중에(v2) 쪼갤 때 앱이 게이트웨이·다중 서비스와 깔끔히 연동되도록 **앱을 먼저 정돈**하는 단계다.

## 3. 정비 5단계 (순서 고정)

| 단계 | 작업 | 목적 | 현재 상태(2026-06-06) | 완료 기준 |
| --- | --- | --- | --- | --- |
| ① | 국제화(다국어) | 문자열을 코드에서 분리, 언어 코드로 전환 | 미설정(`l10n`/`intl`/`.arb` 없음) | `flutter_localizations`+`intl` 셋업, ko/en `.arb`, 주요 화면 문자열 외부화 |
| ② | Riverpod 일관 적용 | 상태관리 일원화 | 의존성·일부 사용, `setState` 11개 잔존 | 앱/비즈니스 상태는 Riverpod로 정리(순수 로컬 UI 상태만 `setState` 허용) |
| ③ | Size·height 변수화 | 하드코딩 수치를 상수로 | 산재 | `core` 상수(dimensions)로 추출·치환 |
| ④ | 컴포넌트화 | 반복 위젯 공통화 | `core/widgets` 일부 | 반복 UI를 공통 컴포넌트로 추출·재사용 |
| ⑤ | 데드코드 정리 | 미사용 코드 제거 | 미정리 | 미사용 import/메서드/파일 제거, `flutter analyze` 무경고 지향 |

## 4. 왜 이게 MSA를 가능하게 하나

- **국제화·변수화·컴포넌트화**로 화면이 데이터·설정 주도로 바뀌면, 백엔드가 여러 서비스로 쪼개져 응답 형태가 달라져도 UI 수정 범위가 작아진다.
- **Riverpod 일관화**로 데이터 출처(어느 서비스/게이트웨이)가 Provider 뒤로 숨겨져, MSA 전환 시 호출 지점 교체가 쉬워진다.
- **데드코드 정리**로 분리 대상 코드가 명확해져, 서비스 경계를 긋기 쉬워진다.

## 5. 작업·검증 방식

- **코드 파일 변경**: Claude가 작성·정적 검토.
- **git(브랜치·커밋·푸시·PR)**: 로컬(T)에서 수행. 권장 브랜치 `refactor/fe-i18n`, `refactor/fe-riverpod` 등 단계별.
- **빌드 검증(로컬 필수)**: `flutter pub get` → `flutter gen-l10n` → `flutter analyze` → 기기/시뮬레이터 확인.
- 단계마다 변경 요약을 남기고, 전체 완료 후 워크플로우/리포트를 정리한다.
