0. 처음 켜면 여기부터

내가 지금 어디에 있는지 모르면 아래 순서만 따라간다.

1단계: 저장소 위치 확인

cd /d C:\workspace\QT-AI-2nd-Team-Project
git status --short --branch
  • ## dev...origin/dev이면 시작 가능하다.
  • 다른 브랜치면 먼저 dev로 이동한다.

2단계: 오늘 할 작업 패널 열기

  • 시뮬레이터 상태 모델 정리면 W1-1
  • simulator UseCase 설계면 W1-2
  • 상태 조회 API 계약 검토면 W1-3
  • 실패·재처리 기준 정리면 W1-4

3단계: workflow 먼저 작성

작업 기록 위치는 별도 구현 GitHub 기준으로 여기다.

workspaces/김태혁/workflows/YYYY-MM-DD_task-name.md
workspaces/김태혁/reports/YYYY-MM-DD_task-name_report.md
중요: 이 문서 저장소에는 실제 Spring Boot/Flutter/OpenAPI 구현 골격을 만들지 않는다. 실제 작업 기록은 별도 구현 저장소의 workspaces/김태혁 기준으로 PR에 적는다.

2. 소유권

내가 마음대로 고칠 수 있는 곳과 협업 확인이 필요한 곳을 구분한다.

OWNER 김태혁 주 담당 경로이 경로는 내가 작업 주도. 그래도 CODEOWNERS에 포함된 협업자는 PR에서 같이 본다.
경로CODEOWNERS 기준작업 예시
domain/study김태혁, 강상민, 강태오시뮬레이터 상태 조회, 해설 조회 경계
domain/ai강상민, 김태혁AI 산출물 생성·검증 결과 연결
flutter-app simulator UI김지민, 김태혁READY 버튼과 팝업 연결
domain/bible/qt이지윤, 김태혁본문 범위와 Today QT 연결
CHECK 다른 사람 확인이 필요한 순간내가 고쳐도 되나? 싶으면 여기 기준으로 판단한다.

강상민 확인

  • 시뮬레이터 산출물이 AI 생성·검증 로그와 닿을 때
  • 실패·재처리 상태 기준이 AI 작업과 연결될 때

이지윤 확인

  • 본문 범위, QT passage, 성경 식별자가 필요할 때
  • Today QT 응답에 상태값을 붙일 때

김지민 확인

  • READY 외 버튼 비활성, 팝업, 빈 상태 문구가 바뀔 때
  • 사용자 화면에서 시뮬레이터 진입점을 바꿀 때

강태오 확인

  • API 계약, 권한, 관리자 트리거, PR 게이트가 바뀔 때
  • study 도메인 경계가 애매할 때

3. 브랜치 생성 방법과 규칙

규칙: 항상 dev에서 새 브랜치를 만든다. master나 dev에 직접 커밋하지 않는다. 브랜치 이름은 {type}/{scope}-{short-description} 형식이다.

처음 브랜치 만들기

cd /d C:\workspace\QT-AI-2nd-Team-Project
git fetch origin
git checkout dev
git pull origin dev
git checkout -b feature/study-simulator-status-api

브랜치 이름 예시

  • feature/study-simulator-status-api — 시뮬레이터 상태 조회
  • feature/study-simulator-clip-query — 클립 저장·조회
  • feature/study-today-qt-status — Today QT 상태 연결
  • test/study-simulator-status — 상태별 테스트
  • bugfix/study-ready-button — READY 버튼 기준 수정
  • fix/study-failed-retry-log — 실패·재처리 로그 수정
주의 브랜치 만들 때 흔한 실수입문자가 가장 자주 막히는 지점

4. PR까지 가는 전체 순서

PR 8단계 PR 루틴작업마다 이 순서대로 반복한다.

1~4단계: 작업 전

  1. dev 최신화
  2. 브랜치 생성
  3. workspaces/김태혁/workflows에 workflow 작성
  4. 작업 패널의 기준 문서 확인

5~8단계: 작업 후

  1. 구현 또는 문서/계약 수정
  2. 검증 명령 실행
  3. workspaces/김태혁/reports에 report 작성
  4. push 후 PR 생성
git status --short
git add qtai-server/src/main/java/com/qtai/domain/study workspaces/김태혁
git commit -m "feat(study): add simulator status flow"
git push -u origin feature/study-simulator-status-api
gh pr create --base dev --head feature/study-simulator-status-api --title "feat(study): add simulator status flow"
CI 자동 검사와 실패 시 행동CI/Claude 리뷰가 실패하면 report에 남기고 고친다.
검사무엇을 보는가실패하면
spring-build서버 빌드/테스트로컬에서 ./gradlew -p qtai-server build
requirements-guard금지 기능/금지 데이터금지 단어와 기능 제거
gitleaks시크릿 노출토큰/비밀번호/키 제거
spectralOpenAPI 문법yaml 계약 수정
claude-pr-review자동 코드 리뷰REQUEST_CHANGES 사유를 반영
