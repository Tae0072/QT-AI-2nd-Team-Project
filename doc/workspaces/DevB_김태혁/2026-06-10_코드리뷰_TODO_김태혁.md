## TODO 2. [P2] qt_video_player tooltip i18n 4건

**문제**: i18n(l10n) 도입 후에도 `flutter-app/lib/features/bible/widgets/qt_video_player.dart`의 4곳에 한글 하드코딩 tooltip이 남음 — `:573` '뒤로', `:802` '일시정지/재생', `:822` '배속', `:845` '전체화면'. 영어 사용자에게 한글이 노출된다.

**해결책**: arb에 키 추가 후 `AppLocalizations` 사용.

```jsonc
// lib/l10n/app_ko.arb 에 추가
"videoBack": "뒤로",
"videoPlay": "재생",
"videoPause": "일시정지",
"videoSpeed": "배속",
"videoFullscreen": "전체화면"
// app_en.arb 에는 "Back" / "Play" / "Pause" / "Speed" / "Fullscreen"
```

```dart
// 예: qt_video_player.dart:573
tooltip: AppLocalizations.of(context).videoBack,
// :802 — 상태 분기
tooltip: value.isPlaying
    ? AppLocalizations.of(context).videoPause
    : AppLocalizations.of(context).videoPlay,
```

- [ ] arb 2개 파일에 키 5종 추가(ko/en) 후 `flutter gen-l10n`(또는 빌드 시 자동 생성) — 생성물 커밋 정책은 기존 lib/l10n 커밋 방식 따름
- [ ] 4곳 교체 + 같은 파일에 다른 잔존 한글 문자열 없는지 1회 스캔
- [ ] 위젯 테스트(`qt_video_player_test.dart`)가 텍스트 의존이면 키 기반으로 보정

**검증**: `flutter analyze` + 기존 테스트 통과, 로케일 en으로 실행해 tooltip 영어 표시 확인.

## TODO 3. [P3] QT영상 캐시 버전/체크섬 검증

**문제**: 현재 캐시는 "같은 클립이면 한 번 받은 파일 재사용" 구조라, 서버에서 클립 원본이 교체돼도(URL 동일) 사용자 기기는 옛 파일을 계속 재생한다.

**해결책(권장 순서)**:

1. **캐시 키에 URL 전체(쿼리 포함) 해시 포함** — 서버가 클립 교체 시 URL(또는 버전 쿼리 `?v=`)을 바꾸는 규칙과 세트로 가장 단순·확실. 기존 캐시 키 sanitization 테스트가 있으니 키 생성 함수만 확장.
2. (서버가 URL 불변을 원하면) API 응답에 `clipVersion`/`contentHash` 필드 추가 → 캐시 메타와 비교해 다르면 재다운로드. 이 경우 서버측 변경은 본인(qt-video 도메인)이 같이 처리.

- [ ] 서버 응답(`QtVideoClip`)에 버전 신호가 이미 있는지 확인 후 1안/2안 결정
- [ ] `QtVideoCache`의 키/검증 로직 수정 + 단위 테스트(키 변경 시 새 파일, 동일 시 재사용)
- [ ] 오래된 캐시 정리(용량 상한 또는 LRU)도 같은 PR에서 가능하면 소규모로

**검증**: 테스트 + dev에서 클립 교체 시나리오 수동 1회(URL 변경 → 재다운로드 확인).
