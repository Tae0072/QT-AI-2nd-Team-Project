# Report 문서 생성 스킬

작업 완료 후 팀 내 표준 Report 문서를 생성한다. 변경 내용, 검증 결과, 금지선 확인, 남은 리스크를 정리하여 작업의 완결성을 보장하는 역할을 한다.

## 템플릿

아래 양식을 **그대로** 사용한다. 각 필드를 실제 작업 결과에 맞게 채운다.

```markdown
# Report — YYYY-MM-DD task-name

| 항목 | 내용 |
| --- | --- |
| 담당자 | 김태혁 |
| 브랜치 | feature/기능-설명-kebab-case |
| 작업 패널 | W주차-번호 작업 제목 |

## 변경 내용
- 실제로 수행한 작업 내용을 구체적으로 기술

## 검증 결과
- `./gradlew -p qtai-server test`: 실행 결과 (PASS/FAIL + 요약)
- `git diff --check`: 실행 결과
- 금지 패턴 검색: 실행 결과

## 확인한 금지선
- AI 자유 챗봇/SSE/RAG 없음
- Kafka/Kubernetes/Helm 없음
- 금지 번역본·가사·음원·직접 YouTube URL 저장 없음
- 담당 범위 밖 변경 없음

## 남은 리스크
- 남은 리스크 또는 없음
```

## 작성 규칙

1. **날짜와 task-name**: Workflow 문서와 동일하게 맞춘다 (같은 작업의 시작/종료 문서 쌍).
2. **변경 내용**: 실제 커밋과 diff를 기반으로 구체적으로 기술한다. "무엇을 왜 변경했는지" 중심으로 작성한다.
3. **검증 결과**: 가능하면 실제로 명령을 실행하고 그 결과를 기록한다. 단순히 "통과"가 아니라 테스트 수, 소요 시간 등 구체적 수치를 포함한다.
4. **금지선**: 프로젝트 내 금지 사항을 하나씩 확인한다. 해당 작업에서 관련 없는 항목이라도 "없음"으로 명시적으로 확인한다.
5. **남은 리스크**: 솔직하게 기술한다. 알려진 제한사항, 추후 작업이 필요한 부분, edge case 등을 포함한다.

## 출력

Report 문서를 마크다운 파일로 저장한다. 파일명은 `report-YYYY-MM-DD-task-name.md` 형식을 따른다. 작업 폴더(워크스페이스)에 저장하고 링크를 제공한다.
