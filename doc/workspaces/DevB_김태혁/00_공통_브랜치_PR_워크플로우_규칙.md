## 1. 적용 범위

이 문서는 별도 구현 GitHub `QT-AI-2nd-Team-Project`에서 적용할 작업 운영 기준이다.

문서 기준 저장소인 `2nd-Team-Project`에는 실제 Spring Boot/Flutter 구현 골격, OpenAPI 파일, CI workflow 파일을 만들지 않는다. `workspaces/` 폴더도 구현 저장소의 작업 기록 기준으로만 정의한다.

### 1.1 실행가이드 HTML 공통 구조 반영 방식

`Lead_강태오_실행가이드.html`의 장점은 개발자가 한 화면에서 역할, 소유권, PR 생성, 일정, AI 에이전트 활용, 자기완결성, 워크플로우, 검증 명령, 금지 패턴을 순서대로 확인할 수 있다는 점이다. 이 구조는 모든 개발자 실행가이드의 공통 UX 기준으로 사용한다.

단, 다운로드본에 포함된 구버전 구현 세부사항은 그대로 복사하지 않는다. 실제 내용은 항상 `07_요구사항_정의서.md` v3.1과 이 문서의 브랜치·PR·작업 기록 기준으로 교체한다.

| 탭 | 유지할 구조 | v3.1 기준 적용 내용 |

## 2. 브랜치 생성 규약

모든 작업은 `dev` 최신화 후 작업 브랜치를 만든다. `dev`와 `master` 직접 push는 금지한다.

```powershell
git checkout dev
git pull origin dev
git checkout -b feature/{scope}-{short-task}
```

| 항목 | 기준 |
| --- | --- |
| 기본 형식 | `{type}/{scope}-{short-task}` |
| PR 대상 | 항상 `dev` |
| 허용 type | `feature`, `bugfix`, `refactor`, `test`, `docs`, `chore`, `hotfix` |
| 브랜치 수명 | 최대 3일 권장. 길어지면 작은 PR로 분리 |
| 담당 범위 밖 변경 | PR 본문에 이유와 리뷰어를 명시 |

| 담당자 | 대표 scope | 브랜치 예시 |
| --- | --- | --- |
| Lead_강태오 | `member`, `qt`, `infra`, `admin` | `feature/qt-today-response`, `feature/member-kakao-oauth`, `chore/infra-quality-gates` |
| 강상민 | `ai` | `feature/ai-explanation-batch` |
| 김지민 | `mobile`, `admin` | `feature/mobile-today-qt-screen`, `feature/admin-dashboard-screen` |
| 김태혁 | `study` | `feature/study-simulator-status-api` |
| 이승욱 | `note`, `sharing`, `praise` | `feature/note-crud`, `feature/sharing-flow` |
| 이지윤 | `bible` | `feature/bible-qt-range-model` |

## 3. 자동 PR 규칙과 방법

여기서 자동 PR은 "PR 생성 후 GitHub Actions와 자동 리뷰가 실행되는 흐름"을 뜻한다. 자동 머지는 아니다.

```powershell
git diff --check
git push -u origin feature/{scope}-{short-task}
gh pr create --base dev --head feature/{scope}-{short-task} --title "feat(scope): short task" --body-file workspaces/{담당자}/reports/YYYY-MM-DD_{task}_report.md
```

| 단계 | 기준 |
| --- | --- |
| 1 | `dev` 최신화 후 작업 브랜치 생성 |
| 2 | 작업 전 `workspaces/{담당자}/workflows/`에 작업 계획 작성 |
| 3 | 구현·문서 수정 후 로컬 검증 실행 |
| 4 | `workspaces/{담당자}/reports/`에 검증 결과와 남은 리스크 작성 |
| 5 | 작업 브랜치 push |
| 6 | `dev` 대상으로 PR 생성 |
| 7 | GitHub Actions, Requirements Guard, Document Guard, 자동 리뷰 결과 확인 |
| 8 | APPROVE와 필수 CI 통과 후 squash merge |

PR 본문에는 아래 항목이 반드시 들어간다.

| 항목 | 기준 |
| --- | --- |
| 변경 범위 | 담당 도메인, 화면, 문서 경로를 명시 |
| 요구사항 근거 | 관련 F-ID 또는 기준 문서 섹션 링크 |
| 검증 명령 | 실제 실행한 명령과 결과 |
| 금지 기준 확인 | AI 자유 챗봇/SSE, Kafka/K8s/RAG, 교회 인증, 금지 번역본 등 위반 없음 |
| 남은 리스크 | 미완료 항목, 후속 PR, 수동 검증 필요 사항 |

### 3.1 PR 템플릿 자동 로드 기준

별도 구현 GitHub의 `.github/pull_request_template.md`에는 아래 블록을 둔다. 개발자별 실행가이드 HTML의 3번 패널은 이 템플릿을 화면에서 바로 복사할 수 있게 보여준다.

| 블록 | 필수 내용 |
| --- | --- |
| 구현 내용 | 무엇을 해결하거나 추가했는지 요약 |
| 관련 이슈/태스크 | 이슈 번호, F-ID, 기준 문서 섹션 |
| 변경 유형 | `feat`, `fix`, `refactor`, `test`, `docs`, `chore` |
| 코드 체크리스트 | v3.1 충돌 없음, 금지 기능/데이터/인프라 없음, 도메인 직접 import 없음 |
| 테스트 체크리스트 | 단위 테스트, 통합 테스트, 미작성 사유 |
| 테스트 방법 | 실제 실행 명령 |
| Workflow/Report | `workspaces/{담당자}/workflows`, `workspaces/{담당자}/reports` 경로 |
| 남은 리스크 | 후속 PR, 수동 검증 필요 사항 |

### 3.2 자동 CI 게이트와 Claude PR 리뷰

자동 PR은 자동 머지가 아니라, PR 생성 후 아래 검사가 자동 또는 반자동으로 실행되는 흐름이다.

| 검사 | 실패 시 | 기준 |
| --- | --- | --- |
| gitleaks | 머지 차단 | Secret 노출 차단 |
| Spectral lint | 머지 차단 | OpenAPI 작성 시 계약 lint |
| 단위 테스트 | 머지 차단 | backend 또는 flutter 변경 범위별 테스트 |
| Docker Compose 검증 | 머지 차단 | v1 배포 기준 확인 |
| Requirements Guard | REQUEST_CHANGES | v3.1 금지 기능·데이터·인프라 차단 |
| Claude PR 리뷰 | REQUEST_CHANGES | 도메인 계약, 금지 패턴, 테스트, 보안, 저작권 리스크, API/화면 정합성, 실패 처리, 운영 로그 확인 |
| 사람 리뷰 | 머지 차단 | 담당자 또는 Lead 승인 |

### 3.3 머지와 Hotfix 기준

기본 머지는 `dev` 브랜치 대상 squash merge다. 필수 CI 통과, Claude/자동 리뷰 이슈 해소, 사람 리뷰 승인 후 머지한다.

긴급 수정도 `dev`에서 `hotfix/{scope}-{short-task}` 브랜치를 만들어 PR로 처리한다. 직접 push로 `dev` 또는 `master`를 수정하지 않는다.

## 4. 개발자별 작업 기록 폴더

별도 구현 GitHub에는 최초 작업 기준 PR에서 아래 폴더를 만든다.

```text
workspaces/
    Lead_강태오/
        README.md
        workflows/
            YYYY-MM-DD_task-name.md
        reports/
            YYYY-MM-DD_task-name_report.md
    강상민/
        README.md
        workflows/
        reports/
    김지민/
        README.md
        workflows/
        reports/
    김태혁/
        README.md
        workflows/
        reports/
    이승욱/
        README.md
        workflows/
        reports/
    이지윤/
        README.md
        workflows/
        reports/
```

| 폴더 | 용도 |
| --- | --- |
| `workflows/` | 작업 전 계획, 담당 파일, 예상 PR 범위, 검증 계획 |
| `reports/` | 작업 후 결과, PR 링크, 검증 명령, CI 결과, 남은 리스크 |
| `README.md` | 담당자 역할, 대표 브랜치 prefix, 주요 담당 경로 |
