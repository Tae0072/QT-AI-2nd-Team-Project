## 2. 고정 기준

| 구분 | 기준 |
| --- | --- |
| Today QT 100% | 시뮬레이터 상태값이 정상 응답하면 100% 기준을 만족한다. 모든 클립이 필수라는 뜻이 아니다. |
| 버튼 처리 | `READY`만 보기 버튼 활성화, `MISSING`/`FAILED`/`DISABLED`는 비활성화 |
| 생성 방식 | 본문 범위별로 클립을 한 번 생성한 뒤 재사용한다. 매 요청 시 재생성하지 않는다. |
| 일반 성경 보기 | 일반 성경 보기에서는 시뮬레이터 진입점을 노출하지 않는다. |
| 도메인 경계 | simulator 도메인은 bible/ai 도메인의 내부 클래스를 직접 import하지 않는다. |
| 실패 처리 | 생성 실패는 `FAILED` 상태와 로그를 남기고 재처리 가능하게 둔다. |

## 5. 브랜치·PR·워크플로우 기록 규칙

| 항목 | 김태혁 기준 |
| --- | --- |
| 브랜치 생성 | `dev` 최신화 후 `feature/simulator-*`, `bugfix/simulator-*`, `test/simulator-*` 형식으로 생성 |
| PR 대상 | 항상 `dev` |
| 자동 PR | 브랜치 push 후 GitHub PR을 생성하면 CI와 자동 리뷰가 실행된다. 자동 머지는 하지 않는다. |
| 작업 폴더 | 별도 구현 GitHub에서 `workspaces/김태혁/workflows`, `workspaces/김태혁/reports`를 사용 |
| 기록 순서 | 작업 전 workflow 작성 → 작업·검증 → report 작성 → PR 생성 |
| 기준 문서 | 상세 명령과 템플릿은 `개발자별_일정표/00_공통_브랜치_PR_워크플로우_규칙.md`를 따른다. |
