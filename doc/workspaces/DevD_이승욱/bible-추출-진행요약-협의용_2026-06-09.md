> 대상: Lead(강태오) · bible 오너 그룹(김지민·이승욱·이지윤) · qt/study 오너(김태혁)
> 작성: DevD 이승욱 · 목적: bible 추출 현황 공유 + **결정·협의 필요 항목** 승인 요청

## 1. 한눈에
bible 도메인을 Strangler 방식으로 독립 `bible-service`로 추출 중. **외부 HTTP 경로는 컷오버 완료**(게이트웨이가 `/api/v1/bible/**`를 bible-service로 라우팅), 모놀리식 내부 소비자 전환을 위한 HTTP 어댑터까지 **기본 비활성(inprocess)** 으로 구현됨. 실제 트래픽 컷오버·DB 분리·모놀리식 제거는 **결정·협의 후** 진행.

진척: 공통 인프라+게이트웨이 인증 100% · **bible 코드 토대 ~85%**(Inc1~Inc3d 준비) · 실 컷오버/DB분리/제거 0%(협의 대기).

## 3. 확립된 패턴 (today-qt·note·sharing·mypage 재사용 템플릿)
- **서비스 스캐폴드**: ai-service 패턴(gated persistence/inbound, 자동설정 exclude)
- **인증**: 게이트웨이 JWT → 신원 헤더 주입 + 서비스 **deny-by-default 필터**(헤더 없으면 401) + **SYSTEM 주체**(배치/캐시 호출은 공유 토큰만) + **토큰 회전 grace window**
- **소비자 전환**: `external/{도메인}` HTTP 어댑터가 api/UseCase 구현 `@Primary`(mode 게이트, 기본 inprocess=무변경), 오류 ErrorCode 역매핑, 재시도
- **운영 진입 체크리스트** 양식(`bible-service-운영진입-체크리스트.md`)

## 4. ⚠ 결정·협의 필요 (승인 요청)
1. **Circuit Breaker 의존성**(Lead/인프라): 모놀리식엔 resilience4j도 spring-cloud BOM도 없음 → CB 도입 시 `spring-cloud-starter-circuitbreaker-resilience4j` 추가 또는 resilience4j 버전 핀(게이트웨이 버전 정합). 비차단(재시도+타임아웃 커버)이라 승인 후 도입.
2. **Inc3d 컷오버**(qt/study 오너 김태혁 + bible 오너): `qtai.bible.client.mode=http` 전환. 소비자(qt: listBooks·범위 / note: 배치) 코드는 인터페이스 의존이라 무변경, **빈 선택만 바뀜**. 배포 시 **양측 토큰 동기화**(`qtai.bible.client.gateway-token` = bible-service `gateway.shared-token`) 필수.
3. **Inc4 DB 분리**(study 오너): `study.glossary_terms.bible_verse_id → bible_verses` **크로스 도메인 FK 제거** 필요(bible 전용 DB로 분리 시). study가 무결성 책임지는 참조값으로 전환.
4. **API 명세 동기화**(Lead): `/api/v1/bible/verses/by-ids` 신규 엔드포인트를 `04_API_명세서.md`에 반영.
