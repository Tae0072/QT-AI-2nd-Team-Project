1. 실제 저장소 지도

이승욱이 자주 만지는 경로만 먼저 외우면 된다.

경로 무엇을 하는 곳인가 이승욱이 할 일
qtai-server/src/main/java/com/qtai/domain/note 노트 도메인. api/client/internal/web 구조. DRAFT, CRUD, 이벤트, 테스트 보강.
qtai-server/src/main/java/com/qtai/domain/sharing 공유/스냅샷 도메인. 현재 Share, ShareSnapshot 골격 있음. 닉네임 나눔, 공유본 스냅샷, 공개 취소 흐름 보강.
qtai-server/src/main/java/com/qtai/domain/praise 찬양 도메인. 현재 Praise, PraiseService 골격 있음. 운영자 큐레이션 조회, 내 찬양 목록 정책 보강.
qtai-server/src/test/java/com/qtai/domain 도메인 테스트. 현재 AI 테스트 중심, note/sharing/praise 테스트는 추가 필요. 작업한 기능의 단위/통합 테스트 추가.
qtai-server/apis/api-v1/openapi.yaml OpenAPI 계약. 현재 paths: {}로 비어 있음. API 계약 PR에서 점진적으로 채운다.
doc/workspaces/DevD_이승욱 이승욱 개인 작업 기록 폴더. 작업 전 workflow, 작업 후 report 작성.
.github/pull_request_template.md PR 본문 체크리스트. PR 만들 때 그대로 채운다.
.github/workflows qt-ai-ci.yml, claude-pr-review.yml. 실패하면 로그를 보고 report에 남긴다.

내가 마음대로 고칠 수 있는 곳과 협업 확인이 필요한 곳을 구분한다.

OWNER 이승욱 주 담당 경로이 경로는 내가 작업 주도. 그래도 CODEOWNERS에 포함된 협업자는 PR에서 같이 본다.
경로CODEOWNERS 기준작업 예시
domain/note이지윤, 김지민, 이승욱, 강태오묵상 DRAFT, 노트 CRUD, 이벤트
domain/sharing김지민, 이승욱, 강태오닉네임 나눔, 댓글/좋아요, 스냅샷

CHECK 다른 사람 확인이 필요한 순간내가 고쳐도 되나? 싶으면 여기 기준으로 판단한다.

이지윤 확인

  • Today QT 식별자, 성경 본문, QT 본문 연결이 필요할 때
  • 노트가 어떤 QT 본문에 붙는지 애매할 때

김지민 확인

  • Flutter 화면, 마이페이지, 내 나눔/내 찬양 화면과 연결될 때
  • 공유 화면 안내 문구나 상태가 바뀔 때

강태오 확인

  • 권한, 회원 상태, 닉네임, 관리자/운영자 API가 바뀔 때
  • PR 게이트나 브랜치 규칙이 애매할 때

강상민 확인

  • AI 산출물과 노트/찬양/나눔 경계가 닿을 때
  • AI 추천처럼 보이는 기능 요청이 들어올 때

3. 브랜치 생성 방법과 규칙

규칙: 항상 dev에서 새 브랜치를 만든다. master나 dev에 직접 커밋하지 않는다. 브랜치 이름은 {type}/{scope}-{short-description} 형식이다.

처음 브랜치 만들기

cd /d E:\workspace\QT-AI-2nd-Team-Project
git fetch origin
git checkout dev
git pull origin dev
git checkout -b feature/note-draft

브랜치 이름 예시

  • feature/note-draft — 묵상 DRAFT
  • feature/note-crud — 노트 CRUD
  • feature/sharing-snapshot — 공유본 스냅샷
  • feature/praise-curation-query — 찬양 큐레이션 조회
  • test/note-draft-idempotency — 테스트 보강
  • fix/sharing-owner-check — 버그 수정
주의 브랜치 만들 때 흔한 실수입문자가 가장 자주 막히는 지점

4. PR까지 가는 전체 순서

PR 8단계 PR 루틴작업마다 이 순서대로 반복한다.

1~4단계: 작업 전

  1. dev 최신화
  2. 브랜치 생성
  3. doc/workspaces/DevD_이승욱/workflows에 workflow 작성
  4. 작업 패널의 기준 문서 확인

5~8단계: 작업 후

  1. 구현 또는 문서/계약 수정
  2. 검증 명령 실행
  3. doc/workspaces/DevD_이승욱/reports에 report 작성
  4. push 후 PR 생성
git status --short
git add qtai-server/src/main/java/com/qtai/domain/note doc/workspaces/DevD_이승욱
git commit -m "feat(note): add meditation draft flow"
git push -u origin feature/note-draft
gh pr create --base dev --head feature/note-draft --title "feat(note): add meditation draft flow"
CI 자동 검사와 실패 시 행동CI/Claude 리뷰가 실패하면 report에 남기고 고친다.
검사무엇을 보는가실패하면
spring-build서버 빌드/테스트로컬에서 ./gradlew -p qtai-server build
requirements-guard금지 기능/금지 데이터금지 단어와 기능 제거
gitleaks시크릿 노출토큰/비밀번호/키 제거

5. 내가 작업해야 하는 체크리스트

작업 패널을 열고 체크박스를 하나씩 완료한다.

W1-1 note/sharing/praise 패키지 위치 정리최상위 도메인 경계 확인

어디서 작업?

  • qtai-server/src/main/java/com/qtai/domain/note
  • qtai-server/src/main/java/com/qtai/domain/sharing
  • qtai-server/src/main/java/com/qtai/domain/praise

어떻게 작업?

  1. api, client, internal, web 구조를 유지한다.
  2. 다른 도메인의 Entity/Service/Repository를 직접 import하지 않는다.
  3. 필요하면 상대 도메인의 api/*UseCase와 DTO만 사용한다.
W1-2 묵상 API 계약 검토현재 TODO 컨트롤러와 04 API 명세 비교
현재 dev의 NoteController, SharingController, PraiseController 주석 경로는 일부 문서 명세와 다를 수 있다. 목표 계약은 doc/프로젝트 문서/04_API_명세서.md와 07_요구사항_정의서.md 기준으로 맞춘다.

확인할 파일

  • domain/note/web/NoteController.java
  • domain/sharing/web/SharingController.java
  • domain/praise/web/PraiseController.java
  • qtai-server/apis/api-v1/openapi.yaml

초보자 작업 순서

  1. 04 API 명세에서 notes/sharing/praise 경로를 검색한다.
  2. 현재 컨트롤러 주석과 실제 목표 경로가 다른지 표로 적는다.
  3. 코드 구현 PR이면 Controller TODO를 명세 기준으로 채운다.
  4. OpenAPI PR이면 paths를 해당 API부터 채운다.
rg -n "/api/v1/notes|/api/v1/sharing-posts|/api/v1/praise-songs" "doc/프로젝트 문서" qtai-server
W1-3 묵상 이벤트 기준 정리ApplicationEventPublisher + 실패 로그

작업 내용

  • 노트 생성/수정/삭제 이벤트 이름 정리
  • 묵상 달력 반영에 필요한 payload 정리
  • 핸들러 실패 로그 필드 정리

완료 기준

  • eventId, 이벤트 타입, 핸들러명, 에러 메시지를 남긴다.
  • 완전 전달 보장이라고 쓰지 않는다.
  • 재처리 가능성을 남긴다.
W2-1 묵상 DRAFT 생성/조회feature/note-draft

어디서 작업?

  • domain/note/api: UseCase, DTO
  • domain/note/internal: Entity, Repository, Service
  • domain/note/web: Controller
  • domain/note/client/qt: QT 도메인 호출 어댑터

작업 순서

  1. DRAFT 조회/생성 UseCase를 정한다.
  2. 같은 사용자+같은 QT 본문 중복 DRAFT를 막는다.
  3. 4개 섹션 저장 기준과 충돌하지 않게 한다.
  4. 테스트를 src/test/java/com/qtai/domain/note에 추가한다.
W2-2 찬양 큐레이션 조회feature/praise-curation-query

어디서 작업?

  • domain/praise/api/ListPraiseUseCase.java
  • domain/praise/internal/PraiseService.java
  • domain/praise/web/PraiseController.java

체크리스트

  • AI 추천이 아니다.
  • 가사/음원/직접 YouTube URL을 저장하지 않는다.
  • 운영자 큐레이션 메타데이터만 조회한다.
  • 저작권 문구는 "저작권 리스크를 낮춘다"로 쓴다.
W3-1 묵상 CRUDfeature/note-crud

작업 순서

  1. 목록, 상세, 수정, 삭제 UseCase를 명확히 나눈다.
  2. 쓰기 Service에는 @Transactional을 붙인다.
  3. 조회는 가능하면 readOnly = true로 둔다.
  4. 삭제 후 같은 QT 노트 재작성 가능성을 확인한다.

완료 기준

  • 작성자 소유권 검증이 있다.
  • Entity를 응답으로 직접 반환하지 않는다.
  • Controller가 Repository를 직접 호출하지 않는다.
W3-2 닉네임 나눔 기본 흐름feature/sharing-snapshot

어디서 작업?

  • domain/sharing/api: Create/Revoke/Get snapshot UseCase
  • domain/sharing/internal: Share, ShareSnapshot, SharingService
  • domain/sharing/client/note: note 도메인 API 호출

체크리스트

  • 저장된 노트만 공유한다.
  • 닉네임 노출 안내가 가능해야 한다.
  • 공유 시점 본문/닉네임/구절 스냅샷을 유지한다.
  • 실명, 이메일, 전화번호를 응답하지 않는다.
W3-3 내 찬양 목록feature/praise-my-list

작업 내용

  1. 사용자별 저장 목록을 분리한다.
  2. 저장/삭제 API 계약을 확인한다.
  3. 저장 대상은 큐레이션 참조 또는 디바이스 참조 기준으로 제한한다.

금지선

  • 가사 저장 금지
  • 음원 서버 저장 금지
  • 직접 YouTube URL 저장 금지
  • AI 찬양 추천 금지

7. 자기완결성 명세

자기완결성은 “다른 사람이 PR만 봐도 무엇을 왜 고쳤고, 어떻게 검증했으며, 남은 리스크가 무엇인지 알 수 있는 상태”다.
DONE 한 작업이 끝났다고 말하려면코드만 있으면 미완성이다.
