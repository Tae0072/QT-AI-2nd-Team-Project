0. 처음 켜면 여기부터

내가 지금 어디에 있는지 모르면 아래 순서만 따라간다.

1단계: 저장소 위치 확인

cd /d C:\workspace\QT-AI-2nd-Team-Project
git status --short --branch
  • ## dev...origin/dev이면 시작 가능하다.
  • 다른 브랜치면 먼저 dev로 이동한다.

2단계: 오늘 할 작업 패널 열기

  • 성경 JSON 후보 출처 정리면 W1-1
  • bible ERD/API 계약 검토면 W1-2
  • QT 본문 범위 모델 검증면 W1-3
  • Today QT bible 응답 기준 정리면 W1-4

3단계: workflow 먼저 작성

작업 기록 위치는 별도 구현 GitHub 기준으로 여기다.

workspaces/이지윤/workflows/YYYY-MM-DD_task-name.md
workspaces/이지윤/reports/YYYY-MM-DD_task-name_report.md
중요: 이 문서 저장소에는 실제 Spring Boot/Flutter/OpenAPI 구현 골격을 만들지 않는다. 실제 작업 기록은 별도 구현 저장소의 workspaces/이지윤 기준으로 PR에 적는다.

2. 소유권

내가 마음대로 고칠 수 있는 곳과 협업 확인이 필요한 곳을 구분한다.

OWNER 이지윤 주 담당 경로이 경로는 내가 작업 주도. 그래도 CODEOWNERS에 포함된 협업자는 PR에서 같이 본다.
경로CODEOWNERS 기준작업 예시
domain/bible이지윤, 강태오성경 본문·출처·범위 조회
domain/qt강태오, 이지윤Today QT 본문 범위
flutter-app bible UI김지민, 이지윤한글 로컬/영어 온라인 화면
domain/note/sharing이승욱, 이지윤@멘션 verseIds 동기화
CHECK 다른 사람 확인이 필요한 순간내가 고쳐도 되나? 싶으면 여기 기준으로 판단한다.

강태오 확인

  • API 계약과 Today QT 응답 조립이 바뀔 때
  • 금지 번역본 검사나 배치 기준이 애매할 때

김지민 확인

  • 한글 번들 다운로드, 오프라인 표시, @멘션 UI가 닿을 때
  • 사용자 화면에서 성경 본문 상태를 표시할 때

이승욱 확인

  • 노트 @멘션 verseIds와 note_verses 저장 기준이 닿을 때
  • 묵상·나눔이 성경 범위를 참조할 때

강상민/김태혁 확인

  • AI 해설 입력이나 시뮬레이터 상태가 bible 범위와 닿을 때
  • Today QT 본문 범위가 산출물 조회 키가 될 때

3. 브랜치 생성 방법과 규칙

규칙: 항상 dev에서 새 브랜치를 만든다. master나 dev에 직접 커밋하지 않는다. 브랜치 이름은 {type}/{scope}-{short-description} 형식이다.

처음 브랜치 만들기

cd /d C:\workspace\QT-AI-2nd-Team-Project
git fetch origin
git checkout dev
git pull origin dev
git checkout -b feature/bible-source-metadata

브랜치 이름 예시

  • feature/bible-source-metadata — 성경 출처 메타데이터
  • feature/bible-verses-range-query — verseFrom/verseTo 범위 조회
  • feature/bible-ko-bundle-api — 한글 번들 다운로드
  • feature/qt-passage-range-model — QT 본문 범위 모델
  • test/bible-forbidden-translations — 금지 번역본 검사
  • fix/bible-offline-ko-message — 오프라인 한글 본문 표시
주의 브랜치 만들 때 흔한 실수입문자가 가장 자주 막히는 지점

4. PR까지 가는 전체 순서

PR 8단계 PR 루틴작업마다 이 순서대로 반복한다.

1~4단계: 작업 전

  1. dev 최신화
  2. 브랜치 생성
  3. workspaces/이지윤/workflows에 workflow 작성
  4. 작업 패널의 기준 문서 확인

5~8단계: 작업 후

  1. 구현 또는 문서/계약 수정
  2. 검증 명령 실행
  3. workspaces/이지윤/reports에 report 작성
  4. push 후 PR 생성
git status --short
git add qtai-server/src/main/java/com/qtai/domain/bible workspaces/이지윤
git commit -m "feat(bible): add source metadata review"
git push -u origin feature/bible-source-metadata
gh pr create --base dev --head feature/bible-source-metadata --title "feat(bible): add source metadata review"
CI 자동 검사와 실패 시 행동CI/Claude 리뷰가 실패하면 report에 남기고 고친다.
검사무엇을 보는가실패하면
spring-build서버 빌드/테스트로컬에서 ./gradlew -p qtai-server build
requirements-guard금지 기능/금지 데이터금지 단어와 기능 제거
gitleaks시크릿 노출토큰/비밀번호/키 제거
spectralOpenAPI 문법yaml 계약 수정
claude-pr-review자동 코드 리뷰REQUEST_CHANGES 사유를 반영
