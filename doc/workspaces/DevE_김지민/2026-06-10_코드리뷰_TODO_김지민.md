## TODO 1. [P1] 로그인 라우팅 수정 — vite proxy 룰 추가

**문제**: 로그인 `POST /api/v1/admin/auth/kakao`는 service-user(8081)에만 있는데, 현재 vite proxy는 모든 `/api`를 admin-server(8090)로 보낸다(`vite.config.ts:14,20-25`). 8090엔 그 컨트롤러가 없고 `/api/v1/admin/**`은 ADMIN 토큰을 요구해 **카카오 키를 넣어도 로그인이 401로 실패**한다. (`LoginPage.tsx:21` 주석의 "백엔드 준비 전"은 사실과 다름 — 서버는 #452로 완료, 라우팅이 원인)

**해결책**: 구체 경로 룰을 `'/api'`보다 **위에** 추가(http-proxy는 정의 순서대로 매칭).

```ts
// vite.config.ts
const proxyTarget = env.VITE_API_PROXY_TARGET ?? 'http://localhost:8090';
const authProxyTarget = env.VITE_AUTH_PROXY_TARGET ?? 'http://localhost:8081';

proxy: {
  // 관리자 카카오 로그인(+추후 refresh)은 service-user 소관 — 8081로
  '/api/v1/admin/auth': { target: authProxyTarget, changeOrigin: true },
  '/api': { target: proxyTarget, changeOrigin: true },
},
```

- [ ] 위 룰 추가 + `.env.example`에 `VITE_AUTH_PROXY_TARGET` 항목·설명 추가
- [ ] `LoginPage.tsx:21` 주석 정정("라우팅 연결 완료, 카카오 JS 키만 주입하면 실로그인 가능"으로)
- [ ] 이승욱과 합동 확인: dev에서 로그인 시 service-user 로그에 요청 도달, 비관리자 403 `ADMIN_USER_NOT_FOUND`·관리자 토큰 발급 확인
- [ ] 참고: 강태오의 nginx 통합안이 확정되면 prod는 nginx가 같은 규칙을 가짐 — 이 vite 룰은 dev 전용으로 그대로 공존 가능
- [ ] 커밋 예: `fix(admin-web): 관리자 인증 경로를 service-user(8081)로 프록시 — 실로그인 복구`

**검증**: dev 서버에서 카카오 로그인 E2E 1회(또는 dev 토큰 경로와 병행 확인).

## TODO 2. [P2] refreshToken 갱신 흐름 추가

**문제**: 로그인 응답의 `refreshToken`(`src/api/adminAuth.ts:13`)을 받기만 하고 안 씀. 401이면 `clearToken()` 후 재로그인 유도(`client.ts:44-46`) → access 30분 정책상 **30분마다 강제 재로그인**.

**해결책**: ① 이승욱과 refresh 엔드포인트 확정(그의 TODO 5에 계약 항목 있음) → ② axios 응답 인터셉터에 single-flight 갱신 구현.

```ts
// client.ts 스케치 — 동시 다발 401에 refresh 1회만 (flutter의 single-flight 패턴과 동일 사상)
let refreshing: Promise<string> | null = null;

async function refreshOnce(): Promise<string> {
  refreshing ??= axios
    .post(`${API_BASE_URL}/admin/auth/refresh`, { refreshToken: getRefreshToken() })
    .then((r) => { saveTokens(r.data.data); return r.data.data.accessToken; })
    .finally(() => { refreshing = null; });
  return refreshing;
}
// 응답 인터셉터: status===401 && !config._retried && refresh 보유 시
//   → config._retried = true; 새 토큰으로 원요청 1회 재시도. 실패 시 clearToken().
```

- [ ] tokenStorage에 refreshToken 저장/삭제 추가(로그아웃 시 함께 제거)
- [ ] `/admin/auth/**` 요청 자체의 401은 재시도 제외(무한루프 방지)
- [ ] 엔드포인트 경로는 이승욱 계약 확정값 사용 — TODO 1의 8081 프록시 룰이 `/api/v1/admin/auth` prefix라 refresh도 같은 prefix면 추가 설정 불필요
- [ ] 간단 테스트(vitest 또는 수동): 401→갱신→재시도 성공 / refresh 실패→로그인 화면

**검증**: access 만료 시나리오(서버에서 짧은 만료로 발급받아 테스트)에서 화면 유지 확인.
