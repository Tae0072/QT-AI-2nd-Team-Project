## 6. 작업 규칙·주의 (중요)

- **브랜치 base = `origin/dev`**, push 후 **dev 대상 PR**.
- **브랜치명**: `feature|bugfix|hotfix|chore|release|docs|test`만 허용(CI `pr-validation`). `feat/`·`fix/`는 **막힘**.
- **검증 게이트(admin-web)**: `cd admin-web && npm run typecheck && npm run build`. (⚠️ admin-web은 CI에 typecheck/build 잡이 **없음** → 로컬 검증이 유일한 안전망. 깨진 채 머지될 수 있으니 항상 직접 검증.)
- **dev 머지**: 항상 `git merge origin/dev` (로컬 `dev`는 origin과 갈라진 stale — 절대 머지 금지).
- `pubspec.lock` transitive 다운그레이드 노이즈는 `git checkout -- flutter-app/pubspec.lock`로 폐기(커밋 금지).
- `qtai-server/Untitled`·`flutter-app/devtools_options.yaml`은 stray — 커밋 제외.
- 설계 선택 있으면 🟢 강 모드(선택지 먼저), 기계적이면 🔵 약 모드(요약·이유 설명) — SKILL 규칙.

## 7. dev 로컬 실행 참고

- admin-web: `cd admin-web && npm run dev` (localhost:5173). API는 vite proxy → admin-server(8090)·인증 service-user(8081). **로컬 admin 로그인은 admin-server(8090) 기동 + 관리자 계정 필요**(현재 어려움 — #510 롤백으로 dev-bypass도 제거됨).
- flutter 웹: `flutter run -d chrome --web-port=3000 --dart-define=WEB_DEV_NO_LOGIN=true --dart-define=WEB_DEV_USER_ID=1` (3000 포트 = 백엔드 CORS 허용). 또는 루트 `run-dev-web.sh`.
