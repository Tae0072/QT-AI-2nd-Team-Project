## PART 1. UseCase 아키텍처 개념

### 1-1. 레이어 4종 (도메인마다 동일)

```
domain.<name>
├─ api/       ← 외부에 공개하는 "계약" = UseCase 인터페이스 + DTO   (메뉴판)
├─ internal/  ← 실제 구현 = Entity·Service·Repository              (부엌)
├─ web/       ← /api/v1/** Controller + Request/Response DTO       (홀/주문창구)
└─ client/    ← 다른 도메인 호출 어댑터 (UseCaseMock 등)            (외부 거래처)
```

- **다른 도메인은 내 `api/`의 인터페이스만 호출 가능.** `internal/`의 Entity·Service·Repository 직접 import 금지(ArchUnit이 막음).
- Controller(web)는 Repository 직접 호출 금지 → 반드시 UseCase(api) 경유.

### 1-2. 요청 데이터 흐름 (예: `POST /api/v1/notes`)

```
[Flutter] note_repository.dart ──POST /api/v1/notes──▶
   │
[web]  NoteController.create(memberId, CreateNoteRequest)
   │     · @AuthenticationPrincipal 로 memberId 추출
   │     · request.toCommand()   ← web DTO → api DTO 변환
   │     · createNoteUseCase.create(memberId, command)   ← 인터페이스 타입으로 호출
   ▼
[api]  CreateNoteUseCase.create(...)   ← 계약만 (구현 없음)
   ▼
[internal] NoteService.create(...)     ← 실제 구현
   │     · normalize() → validateForSave() → Note.create()
   │     · noteRepository.saveAndFlush()  (DB)
   │     · replaceNoteVerses() / publishJournalEvent()
   │     · return NoteCreateResponse (api DTO)
   ▼
[web]  ApiResponse.success(...) 로 감싸 JSON 응답
```

**기억할 점**
- DTO 2종: `CreateNoteRequest`(web 전용) → `CreateNoteCommand`(api, 도메인 간 전달). Entity는 절대 그대로 노출 안 함.
- 도메인 경계 호출: `getBibleVerseUseCase`(bible), `noteQtClient`(qt) 처럼 **남의 api 인터페이스만** 사용.

### 1-3. "인터페이스를 호출했는데 어떻게 맞는 구현이 실행돼?" → Spring DI

```java
// NoteController
private final CreateNoteUseCase createNoteUseCase;  // 인터페이스 타입만 선언

// NoteService
@Service
public class NoteService implements ListNotesUseCase, GetNoteUseCase,
        CreateNoteUseCase, UpdateNoteUseCase, DeleteNoteUseCase, ListNoteCategoriesUseCase { ... }
```

- 서버 기동 시 Spring이 `@Service` Bean(`NoteService`) 1개 생성.
- Controller가 `CreateNoteUseCase` 타입을 요구 → Spring이 "그 인터페이스 구현한 Bean = NoteService" 자동 주입(DI).
- `createNoteUseCase.create()` 호출 → 다형성으로 **실제 `NoteService.create()` 실행.**
- **내가 메서드를 맞추는 게 아니라, 타입(인터페이스)이 같으면 Spring/자바가 알아서 연결.**

> 핵심: **하나의 Service가 여러 UseCase를 동시에 implements** 가능. (NoteService = 6개)

### 1-4. 4가지 질문 정리

| 질문 | 답 |
|---|---|
| 1 API = 1 UseCase? | 정확히 1:1 강제는 아님. **"하나의 의도 = 인터페이스 1개"**, 그 안에 관련 메서드 여러 개 가능(`GetNoteUseCase` = `get`+`getDraft`). 우리 팀은 읽기/생성/수정/삭제를 잘게 쪼갠 스타일(ISP). |
| 인터페이스→구현 연결? | Spring DI + 다형성이 자동 처리. |
| Service가 UseCase 메서드만? | 아님. **`@Override`(공개 계약) + `private`(내부 헬퍼)** 둘 다 보유가 정상. |
| api에 안 올리는 메서드? | `private` 보조 메서드(`normalize`, `validateForSave` 등)는 구현 세부라 api 인터페이스에 안 올림 = 올바름. |

## PART 4. 작업 시 체크리스트 (내 규칙)

- [ ] 새 기능 = "외부 공개냐?" → 공개면 `api/`에 UseCase 인터페이스 + DTO 추가, 내부 전용이면 `internal/`만.
- [ ] Service는 `implements <UseCase>` + `@Override` 공개 메서드 + `private` 헬퍼.
- [ ] Controller는 **인터페이스 타입만** 주입, Repository 직접 호출 금지.
- [ ] web DTO(`...Request`) → api DTO(`...Command`) 변환(`toCommand()`). Entity 직접 노출 금지.
- [ ] 다른 도메인 호출은 **상대 `api/UseCase`** 만. 통합 전엔 `client/{도메인}/...UseCaseMock`, 진짜 구현 등록되면 Mock 삭제.
- [ ] write/command 메서드 `@Transactional`, read 메서드 `@Transactional(readOnly = true)`.
- [ ] 기능엔 F-ID 남기기 (note=F-03/F-16, sharing=F-10).
