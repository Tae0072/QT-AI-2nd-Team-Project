## 1. Entity — `BibleVerse.java`

**파일 경로**: `qtai-server/src/main/java/com/qtai/domain/bible/internal/BibleVerse.java`

이 파일은 아직 TODO 골격이지만, 어떤 어노테이션이 들어갈지가 주석으로 모두 적혀 있어 학습용으로 적합하다.

### 1-1. 패키지 위치: `domain.bible.internal`

```java
package com.qtai.domain.bible.internal;
```

- 자바의 `package`는 폴더 경로와 같다.
- 끝의 `.internal` 이 핵심. `CLAUDE.md §4` 에서 정한 규칙:
  - Entity, Service, Repository, QueryRepository, 도메인 전용 예외는 모두 `internal` 패키지 안에 둔다.
  - **다른 도메인은 `internal`을 절대 import 할 수 없다.**
  - 외부 도메인이 사용하려면 반드시 `api/` 패키지의 UseCase 인터페이스를 거쳐야 한다.

> 적용: 김지민이 작성할 `note/internal/NoteService` 도 sharing 도메인에서 직접 import하면 안 된다. sharing은 `note/api/GetNoteUseCase` 인터페이스를 통해 호출한다.

### 1-2. `@Entity`, `@Table`, `uniqueConstraints`

```java
@Entity
@Table(name = "bible_verse",
       uniqueConstraints = @UniqueConstraint(
           columnNames = {"book","chapter","verse","translation"}))
public class BibleVerse { ... }
```

- `@Entity`: "이 클래스는 DB 테이블 한 행을 표현한다"는 선언. JPA(자바의 ORM)가 객체 ↔ 테이블 매핑을 자동으로 처리.
- `@Table(name = ...)`: 실제 DB 테이블 이름 지정. 클래스 이름과 다르게 둘 수 있다.
- `uniqueConstraints`: 지정한 컬럼 조합이 중복되면 DB 차원에서 거부한다는 무결성 규칙.

> 적용: 김지민이 다룰 `notes` 테이블에도 `active_unique_key` 로 중복 방지가 걸려 있다(요구사항 정의서). 같은 개념이다.

### 1-3. `@Id`, `@GeneratedValue`

```java
@Id
@GeneratedValue
Long id;
```

- `@Id`: 이 필드가 Primary Key(테이블의 고유 식별자) 임을 표시.
- `@GeneratedValue`: 새 행이 생길 때 DB가 자동으로 1, 2, 3... 증가하는 값을 채워준다. 코드에서 직접 id를 만들 필요 없음.

### 1-4. `@Column(columnDefinition="TEXT")`

```java
@Column(columnDefinition="TEXT") String text;
```

- 기본 매핑은 `VARCHAR(255)` 이지만, 본문처럼 긴 문자열은 `TEXT` 타입으로 명시한다.
- 길이 제한이 사실상 없어서 본문·노트 내용 같은 긴 글에 사용.

> 적용: 노트의 `content` 도 같은 이유로 `TEXT` 가 된다.

### 1-5. 핵심 용어 정리표 (1번 파일에서 등장)

| 용어 | 한 줄 요약 |
| --- | --- |
| `package ...internal` | 다른 도메인이 import 못 하는 비공개 영역 |
| `@Entity` | "이 클래스 = DB 테이블 한 행" 선언 |
| `@Table(uniqueConstraints=...)` | 테이블 이름 + 컬럼 조합 중복 방지 |
| `@Id` | Primary Key 표시 |
| `@GeneratedValue` | DB가 자동으로 PK 값을 채워줌 |
| `@Column(columnDefinition="TEXT")` | 긴 문자열 컬럼 타입 지정 |
| JPA | 자바 객체와 DB 테이블을 자동 매핑해주는 표준 (ORM) |
| ORM | Object-Relational Mapping. 객체-테이블 변환 기술의 총칭 |

## 2. Repository — `BibleRepository.java`

**파일 경로**: `qtai-server/src/main/java/com/qtai/domain/bible/internal/BibleRepository.java`

Entity가 "테이블 모양"이라면 Repository는 "그 테이블에 접근하는 방법"이다. Spring Data JPA를 사용하면 개발자가 SQL을 거의 쓰지 않아도 된다.

### 2-1. 클래스가 아니라 인터페이스

```java
public interface BibleRepository { ... }
```

- Repository는 항상 `interface` 로 선언한다.
- Spring Data JPA가 **런타임에 구현체를 자동 생성** 한다.
- 개발자가 직접 `INSERT`, `SELECT` SQL을 쓸 필요가 없다.

### 2-2. `JpaRepository` 상속으로 기본 기능 획득

```java
public interface BibleRepository extends JpaRepository<BibleVerse, Long> { ... }
```

상속만 받으면 다음 메서드가 모두 사용 가능해진다.

| 메서드 | 역할 |
| --- | --- |
| `save(entity)` | INSERT 또는 UPDATE |
| `findById(id)` | PK로 한 건 조회 |
| `findAll()` | 전부 조회 |
| `deleteById(id)` | 삭제 |
| `count()` | 총 개수 |

제네릭 `<BibleVerse, Long>` 의 두 자리:
- 첫째: 다룰 Entity 클래스
- 둘째: 그 Entity의 PK 타입

> 적용: 김지민의 노트 Repository는 `extends JpaRepository<Note, Long>` 가 된다.

### 2-3. 메서드 이름으로 쿼리 자동 생성 (Query Method)

```java
Optional<BibleVerse> findByBookAndChapterAndVerseAndTranslation(
    String book, int chapter, int verse, String translation);
```

메서드 이름 자체가 SQL이 된다. Spring Data JPA가 다음을 자동 생성:

```sql
SELECT * FROM bible_verse
WHERE book = ? AND chapter = ? AND verse = ? AND translation = ?
```

규칙:
- `findBy` 뒤에 컬럼명을 `And` 로 이어 붙이면 WHERE 절이 됨
- 메서드 파라미터 순서가 컬럼 순서와 일치해야 함

자주 쓰는 키워드:

| 키워드 | 의미 |
| --- | --- |
| `And`, `Or` | 조건 결합 |
| `Between` | `BETWEEN ? AND ?` |
| `LessThan`, `GreaterThan` | `< ?`, `> ?` |
| `Like` | `LIKE ?` (와일드카드 검색) |
| `In` | `IN (?)` |
| `IsNull`, `IsNotNull` | NULL 검사 |
| `OrderBy필드명Desc` | 정렬 |

### 2-4. `Optional<T>` — null 대신 명시적 부재 표현

```java
Optional<BibleVerse> findBy...
```

- 자바에서 "있을 수도, 없을 수도 있는 값" 을 표현하는 방식.
- `null` 을 그대로 반환하지 않으므로 NullPointerException 위험을 컴파일러가 잡아준다.

사용 예:

```java
BibleVerse verse = bibleRepository.findById(1L)
    .orElseThrow(() -> new NotFoundException("절을 찾을 수 없습니다"));
```

### 2-5. `Between` 으로 범위 조회

```java
List<BibleVerse> findByBookAndChapterAndVerseBetweenAndTranslation(
    String book, int chapter, int start, int end, String translation);
```

→ 자동 생성: `WHERE book = ? AND chapter = ? AND verse BETWEEN ? AND ? AND translation = ?`

### 2-6. 복잡한 쿼리 — `@Query` 와 `Page<T>`

```java
@Query(nativeQuery=true)
Page<BibleVerse> ...
```

- 메서드 이름으로 표현하기 어려운 쿼리는 `@Query` 에 SQL을 직접 쓴다.
- `nativeQuery=true` 는 MySQL 문법을 그대로 사용한다는 뜻 (예: FULLTEXT MATCH).
- 반환 타입 `Page<T>` 는 결과 데이터 + 전체 개수 + 페이지 정보를 함께 담은 페이지네이션 응답이다.

> 적용: `GET /api/v1/sharing-posts` (PUBLISHED + 페이지네이션) 에서 `Page<SharingPost>` 를 반환하게 된다.

### 2-7. 핵심 용어 정리표 (2번 파일에서 등장)

| 용어 | 한 줄 요약 |
| --- | --- |
| `interface` | 메서드 시그니처만 선언, 구현은 다른 곳에 |
| Spring Data JPA | 인터페이스 선언만으로 구현체를 자동 생성해주는 라이브러리 |
| `JpaRepository<T, ID>` | save/findById/findAll 등 기본 CRUD 메서드를 제공 |
| Query Method | 메서드 이름을 분석해 SQL을 자동 생성하는 기능 |
| `Optional<T>` | "값이 있을 수도 없을 수도" 를 표현하는 자바 표준 타입 |
| `Between` | 범위 조회 키워드 |
| `@Query` | 메서드 이름 규칙으로 어려운 쿼리를 직접 작성할 때 사용 |
| `nativeQuery=true` | DB 고유 문법(MySQL 등)을 그대로 사용 |
| `Page<T>` | 페이지네이션 응답 (데이터 + 전체 개수 + 페이지 정보) |
| FULLTEXT 인덱스 | MySQL의 전문 검색 인덱스. 본문에서 키워드 빠르게 찾기 |

## 3. Service — `BibleService.java`

**파일 경로**: `qtai-server/src/main/java/com/qtai/domain/bible/internal/BibleService.java`

Service는 비즈니스 로직이 들어가는 자리. 김지민이 가장 많이 작성하게 될 코드 종류다. 골격에 7개의 핵심 어노테이션·패턴이 모두 등장한다.

### 3-1. `@Service` — Spring 빈으로 등록

```java
@Service
public class BibleService { ... }
```

- Spring에게 "이 클래스는 비즈니스 로직 담당, 네가 관리하는 객체로 등록해줘" 라고 알리는 표시.
- Spring이 관리하는 객체를 **빈(Bean)** 이라고 부른다.
- 빈이 되면:
  - 다른 클래스가 필요할 때 Spring이 자동으로 주입 (의존성 주입, DI)
  - Spring이 트랜잭션·캐시·트랜잭션 롤백 등 부가 기능을 자동으로 적용

| 어노테이션 | 역할 |
| --- | --- |
| `@Service` | 비즈니스 로직 계층 |
| `@Repository` | 데이터 접근 계층 |
| `@Controller` / `@RestController` | HTTP 입구 |
| `@Component` | 일반 빈 (위 3개는 의미만 다른 특수화) |

### 3-2. `@RequiredArgsConstructor` (Lombok)

```java
@RequiredArgsConstructor
public class BibleService {
    private final BibleRepository bibleRepository;
}
```

이 한 줄이 다음 코드를 자동 생성한다.

```java
public BibleService(BibleRepository bibleRepository) {
    this.bibleRepository = bibleRepository;
}
```

- Lombok은 반복 코드(생성자, getter, setter 등)를 어노테이션으로 대신 만들어주는 라이브러리.
- `@RequiredArgsConstructor` 는 `final` 이 붙은 필드만 받는 생성자를 만든다.

### 3-3. 생성자 주입 + `final` (의존성 주입)

```java
private final BibleRepository bibleRepository;
```

- `final` = 한 번 값이 들어오면 절대 바뀌지 않음.
- Spring이 객체를 만들 때 생성자를 통해 `BibleRepository` 빈을 꽂아준다 → 이후 변경 불가.
- **권장되는 DI 방식**. `@Autowired` 필드 주입보다 안전하고 테스트하기 쉽다.

| 용어 | 의미 |
| --- | --- |
| 의존성 주입 (DI) | 객체가 필요로 하는 다른 객체를 직접 만들지 않고, 외부(Spring)에서 받아오는 방식 |
| 생성자 주입 | 생성자 파라미터로 의존성을 받는 방식 (권장) |
| 필드 주입 | `@Autowired` 를 필드에 붙이는 방식 (지양) |

### 3-4. `implements UseCase` — 도메인 경계의 핵심

```java
public class BibleService implements GetBibleVerseUseCase, ListChaptersUseCase, SearchBibleUseCase { }
```

이 한 줄이 CLAUDE.md §3, §4 도메인 경계 규칙의 핵심이다.

- **UseCase** (인터페이스, `api/` 패키지) = 외부 도메인에 약속한 "공개 기능 목록"
- **Service** (구현체, `internal/` 패키지) = 그 약속의 실제 구현
- 다른 도메인은 UseCase 타입만 알고, Service 구조는 모른다.

도메인 간 호출 흐름:

```
sharing 도메인 ── 호출 ──> note/api/GetNoteUseCase 인터페이스
                                       ▲
                                       │ implements
                                       │
                              note/internal/NoteService
```

> 효과: Service 내부 구현을 바꿔도 다른 도메인 코드를 손댈 필요 없다. 도메인 간 결합도가 낮아진다.

### 3-5. `@Transactional` — 트랜잭션 경계

```java
@Transactional(readOnly = true)
public class BibleService { ... }
```

- 트랜잭션 = "이 메서드 안에서 일어나는 DB 작업들은 하나의 묶음. 중간에 실패하면 전부 ROLLBACK."
- `readOnly = true` 는 "이 안에서는 SELECT만 함, INSERT/UPDATE 없음" 선언 → JPA가 더 빠르게 동작.
- 클래스 위에 붙이면 모든 메서드의 기본값이 된다.
- 데이터 변경 메서드는 메서드 위에 `@Transactional` 만 붙여서 readOnly를 덮어쓴다.

> CLAUDE.md §9: write use case와 command service에는 `@Transactional` 을, read path에는 `@Transactional(readOnly = true)` 를 적용.

```java
@Transactional(readOnly = true)  // 클래스 기본값
public class NoteService implements ... {

    public NoteResponse getNote(Long id) { ... }            // 기본값 적용 (readOnly)

    @Transactional                                            // 덮어쓰기 (쓰기 가능)
    public Long createNote(NoteCreateRequest request) { ... }
}
```

### 3-6. `@Cacheable` — 메서드 결과 캐싱

```java
@Cacheable("bible-books")
public List<String> listBooks() { ... }
```

- "같은 입력이면 메서드 본문을 실행하지 않고 캐시에서 결과 반환."
- 자주 호출되지만 거의 변하지 않는 데이터에 사용.
- 캐시 이름(`"bible-books"`) 으로 캐시 영역을 구분.
- 캐시 저장소는 `application.yml` 설정에 따라 Caffeine(인메모리) 또는 Redis가 된다.

### 3-7. 공통 예외 사용

```java
throw new BusinessException(INVALID_INPUT);
```

- 프로젝트 공통 예외를 던지면 공통 예외 핸들러가 받아서 클라이언트에게 동일한 형식의 에러 응답을 반환한다.
- 각 메서드에서 응답 JSON을 직접 만들지 않아도 된다.

> 이승욱이 만들 공통 에러 코드(`DUPLICATE_NOTE`, `TOKEN_EXPIRED`, `FORBIDDEN` 등)가 여기 들어간다.

### 3-8. 핵심 용어 정리표 (3번 파일에서 등장)

| 용어 | 한 줄 요약 |
| --- | --- |
| `@Service` | "비즈니스 로직 빈으로 등록" 표시 |
| Bean (빈) | Spring이 관리하는 객체 |
| Lombok | 반복 코드를 어노테이션으로 대신 만들어주는 라이브러리 |
| `@RequiredArgsConstructor` | `final` 필드를 받는 생성자 자동 생성 |
| DI (의존성 주입) | 필요한 객체를 외부에서 받아오는 방식 |
| 생성자 주입 | 생성자 파라미터로 의존성을 받는 권장 방식 |
| `final` 필드 | 한 번 값이 들어오면 변경 불가 |
| `implements UseCase` | 외부 공개 계약(인터페이스)의 실제 구현 표시 |
| `@Transactional` | DB 작업을 한 묶음으로 처리, 실패 시 전부 ROLLBACK |
| `readOnly = true` | 읽기 전용 트랜잭션, 성능 향상 |
| ROLLBACK | DB 변경을 되돌리는 동작 |
| `@Cacheable` | 메서드 결과를 캐시에 저장 |
| `BusinessException` | 프로젝트 공통 예외 |

## 4. DTO — `BibleVerseResponse.java`, `BibleSearchRequest.java`

**파일 경로**:
- `qtai-server/src/main/java/com/qtai/domain/bible/api/dto/BibleVerseResponse.java`
- `qtai-server/src/main/java/com/qtai/domain/bible/api/dto/BibleSearchRequest.java`

DTO(Data Transfer Object)는 계층 간·시스템 간 데이터 전달용 객체. 김지민이 가장 많이 만들 클래스 종류다. API 하나당 보통 Request + Response 한 쌍이 생긴다.

### 4-1. `record` — Java 21의 불변 데이터 클래스

```java
public record BibleVerseResponse(Long id, String book, Integer chapter, ...) {}
```

`record` 는 데이터만 담는 클래스를 한 줄로 만드는 자바 14+ 기능. Java 21 표준.

위 한 줄로 자동 생성되는 것:

| 항목 | 예시 |
| --- | --- |
| 생성자 | `new BibleVerseResponse(1L, "요한복음", 3, ...)` |
| 게터 | `verse.book()`, `verse.chapter()` — `get` 없이 필드명만 사용 |
| `equals()` | 모든 필드 값으로 비교 |
| `hashCode()` | 모든 필드 기반 |
| `toString()` | 모든 필드 출력 |
| 모든 필드 `final` | 한 번 만들면 못 바꿈 (불변, Immutable) |

→ DTO는 "데이터만 들고 다니면 되는 객체" 라서 `record` 와 궁합이 완벽하다.

### 4-2. `api.dto` 패키지 — 외부 공개 영역

```java
package com.qtai.domain.bible.api.dto;
```

- `internal` 이 아닌 **`api.dto`** 패키지 → 외부 도메인이 import 가능.
- DTO는 외부 계약(API)의 일부이기 때문에 여기 둔다.

> 적용 위치: `note/api/dto/NoteCreateRequest.java`, `note/api/dto/NoteResponse.java`, `sharing/api/dto/SharingPostResponse.java` 등.

### 4-3. DTO를 따로 만드는 이유 (Entity와 분리)

Entity를 Controller가 그대로 반환하면 안 되는 이유:

| 문제 | DTO가 해결하는 방법 |
| --- | --- |
| Entity에 민감 필드(비밀번호 해시 등)가 있음 | DTO에는 안전한 필드만 담음 |
| Entity 구조가 바뀌면 API 응답도 바뀜 | DTO를 사이에 두면 Entity가 바뀌어도 API는 유지 가능 |
| 한 Entity로 여러 화면을 만들어야 함 | 화면마다 다른 DTO를 만든다 |
| JPA 연관관계 때문에 직렬화 시 무한 루프 발생 | DTO는 단순한 값 객체라 안전 |

> 원칙: **Controller는 절대 Entity를 그대로 반환하지 않는다.** 항상 DTO로 변환해서 반환한다.

### 4-4. Request DTO — 검증 어노테이션 (Bean Validation)

```java
public record BibleSearchRequest(
    @NotBlank String keyword,           // null 또는 빈 문자열이면 자동 거부
    String translation,                 // 검증 없음 (nullable)
    String book,
    Integer chapter
) {}
```

자주 쓰는 검증 어노테이션:

| 어노테이션 | 의미 |
| --- | --- |
| `@NotNull` | null 금지 |
| `@NotBlank` | null + 빈 문자열 + 공백만 있는 문자열 금지 (String 전용) |
| `@NotEmpty` | null + 빈 컬렉션/문자열 금지 |
| `@Size(min=1, max=100)` | 길이·크기 제한 |
| `@Min(0)`, `@Max(100)` | 숫자 범위 |
| `@Email` | 이메일 형식 |
| `@Pattern(regexp = "...")` | 정규식 매칭 |

- Controller 파라미터에 `@Valid` 를 함께 붙이면 Spring이 자동 검증.
- 검증 실패 시 자동으로 `400 Bad Request` 응답 반환 → Controller에서 if 문으로 일일이 체크 안 해도 됨.

> 적용: 김지민의 `NoteCreateRequest` 예시
> ```java
> public record NoteCreateRequest(
>     @NotBlank String title,
>     @NotBlank String content,
>     @NotNull NoteCategory category
> ) {}
> ```

### 4-5. Request vs Response 역할 분리

| | 방향 | 위치 | 검증 어노테이션 |
| --- | --- | --- | --- |
| Request DTO | 외부 → 서버 | Controller 메서드 파라미터 | 사용 (`@NotBlank` 등) |
| Response DTO | 서버 → 외부 | Controller 메서드 반환값 | 없음 (이미 검증된 데이터) |

같은 도메인에서도 동작별로 DTO를 분리한다.

| 동작 | DTO 예시 | 특징 |
| --- | --- | --- |
| 생성 요청 | `NoteCreateRequest` | id 없음, title/content 필수 |
| 단건 응답 | `NoteResponse` | id 포함, 생성 시간 포함 |
| 수정 요청 | `NoteUpdateRequest` | 변경 가능한 필드만 |
| 목록 응답 | `NoteListResponse` 또는 `Page<NoteResponse>` | 페이지 정보 포함 |

→ 한 도메인에 DTO가 5~10개 생기는 게 정상이다.

### 4-6. 핵심 용어 정리표 (4번 파일에서 등장)

| 용어 | 한 줄 요약 |
| --- | --- |
| DTO (Data Transfer Object) | 계층 간·시스템 간 데이터 전달용 객체 |
| `record` | Java 14+의 불변 데이터 클래스 선언 키워드 |
| 불변 객체 (Immutable) | 생성 후 값을 바꿀 수 없는 객체 |
| Bean Validation | 자바 표준 검증 어노테이션 모음 (`@NotBlank` 등) |
| `@Valid` | Controller 파라미터에 붙여 검증을 활성화 |
| Request DTO | 외부에서 들어오는 입력 객체 |
| Response DTO | 외부로 나가는 출력 객체 |
| 직렬화 (Serialization) | 객체를 JSON 등으로 변환하는 과정 |

## 5. UseCase — `GetBibleVerseUseCase.java`, `ListChaptersUseCase.java`, `SearchBibleUseCase.java`

**파일 경로**:
- `qtai-server/src/main/java/com/qtai/domain/bible/api/GetBibleVerseUseCase.java`
- `qtai-server/src/main/java/com/qtai/domain/bible/api/ListChaptersUseCase.java`
- `qtai-server/src/main/java/com/qtai/domain/bible/api/SearchBibleUseCase.java`

도메인 경계의 가장 중요한 자리. 외부 도메인이 이 도메인에 접근할 수 있는 유일한 통로다.

### 5-1. 항상 인터페이스 (`interface`)

```java
public interface GetBibleVerseUseCase {
    BibleVerseResponse getVerse(Long verseId);
}
```

UseCase는 항상 `interface`. 메서드 시그니처(이름·파라미터·반환 타입)만 있고 구현 본문은 없다.

| 이유 | 설명 |
| --- | --- |
| 도메인 간 결합도 ↓ | 호출자는 인터페이스만 보고 호출. 내부 구현은 몰라도 됨 |
| 구현 교체 자유 | Service 내부 로직을 바꿔도 호출자 코드 변경 0 |
| 테스트가 쉬움 | `client/{타도메인}/...UseCaseMock.java` 같은 가짜 구현체 만들기 쉬움 |
| 순환 의존 방지 | 도메인 간 직접 import가 아니라 인터페이스 의존 → 사이클 발생 시 컴파일러가 잡음 |

### 5-2. `api` 패키지 위치

```java
package com.qtai.domain.bible.api;
```

- `internal` 이 아닌 **`api`** → 외부 도메인이 import 가능한 공개 영역.
- CLAUDE.md §4: "다른 도메인 호출은 항상 상대 도메인의 `api/UseCase` 인터페이스를 통한다."

### 5-3. 반환 타입은 반드시 DTO

```java
BibleVerseResponse getVerse(Long verseId);   // ✅ DTO 반환
BibleVerse        getVerse(Long verseId);     // ❌ Entity 반환 금지
```

UseCase 시그니처에 Entity가 등장하면 외부 도메인이 Entity 구조에 의존하게 된다. 반드시 `api/dto/` 의 DTO만 사용한다.

> CLAUDE.md §3: "다른 도메인의 Entity, Service, Repository, infrastructure 타입을 직접 import하지 않는다."

### 5-4. 인터페이스 분리 원칙 (ISP)

bible 도메인은 한 개의 큰 인터페이스 대신 3개로 쪼개져 있다.

| 인터페이스 | 메서드 수 | 용도 |
| --- | --- | --- |
| `GetBibleVerseUseCase` | 2개 (`getVerse`, `getVerses`) | 절 단건/범위 조회 |
| `ListChaptersUseCase` | 2개 (`listBooks`, `listChapters`) | 책·장 목록 |
| `SearchBibleUseCase` | 1개 (`search`) | 키워드 검색 |

**Interface Segregation Principle (ISP)**:
- 검색만 필요한 도메인은 `SearchBibleUseCase` 만 의존
- 책 목록만 필요한 도메인은 `ListChaptersUseCase` 만 의존
- 안 쓰는 메서드까지 끌고 다니지 않음 → 변경 영향 최소화

> 적용: 김지민의 `note/api/` 도 동작별로 쪼갠다. 이미 골격에 다음이 있음.
> - `CreateNoteUseCase`
> - `GetNoteUseCase`
> - `UpdateNoteUseCase`
> - `DeleteNoteUseCase`

### 5-5. 호출자(consumer)를 주석에 명시

```java
/**
 * 성경 절 조회 UseCase 포트.
 * 호출자: qt / study / sharing 도메인.
 */
```

UseCase를 정의할 때 "누가 부를 건지" 를 먼저 정한다. 이유:
- 호출자가 정해지면 그 도메인이 필요한 데이터 모양으로 시그니처를 짤 수 있음.
- 호출자가 없는 UseCase는 만들 이유가 없음.

> 적용: 김지민의 `note/api/GetNoteUseCase` 는 "sharing 도메인이 노트 스냅샷을 만들 때 부른다" 가 호출자다. 이 사실이 정해져야 어떤 필드를 응답 DTO에 담을지 결정할 수 있다.

### 5-6. 헥사고날 아키텍처 용어 — "포트(Port)"

주석에 "UseCase 포트" 라는 표현이 등장한다. 헥사고날 아키텍처 용어.

| 용어 | 의미 |
| --- | --- |
| 포트 (Port) | 외부와 통하는 약속(인터페이스) |
| 어댑터 (Adapter) | 포트를 구현한 실제 클래스 (Service, Client 등) |

도메인은 포트(인터페이스) 만 알고, 어댑터(구현)는 외부에서 꽂아주는 구조. CLAUDE.md §4의 `client/` 패키지가 "외부 시스템을 우리 포트에 어댑팅하는 자리".

5개월 차에서는 깊이 파지 말고, **"포트 = 인터페이스, 외부와 통하는 약속"** 정도로 받아들이면 충분.

### 5-7. 도메인 간 호출 흐름 (전체 그림)

```
[ sharing 도메인 ]
        │
        │  스냅샷 만들 때 노트 본문 필요
        │
        │  ┌── note/api/GetNoteUseCase (interface, 포트)
        ▼  ▼
    NoteResponse getNote(Long id);
                    ▲
                    │ implements
                    │
        ┌── note/internal/NoteService (어댑터, 실제 구현)
        │   - Entity ↔ DTO 변환
        │   - DB 조회
        │   - 비즈니스 로직
        └─────────────────────────────────────────────
```

- sharing은 `note.internal` 을 절대 import하지 않는다.
- sharing은 `note.api.GetNoteUseCase` 와 `note.api.dto.NoteResponse` 만 import한다.
- 통합 전에는 `sharing/client/note/GetNoteUseCaseMock.java` 로 임시 구현하여 작업 (이미 골격에 존재).

### 5-8. 핵심 용어 정리표 (5번 파일에서 등장)

| 용어 | 한 줄 요약 |
| --- | --- |
| UseCase | 외부에 공개하는 도메인 기능 약속 (인터페이스) |
| `api` 패키지 | UseCase와 DTO가 모이는 공개 영역 |
| ISP (Interface Segregation Principle) | 인터페이스 분리 원칙. 동작별로 작게 쪼갠다 |
| 호출자 (Consumer) | UseCase를 호출하는 다른 도메인 |
| 포트 (Port) | 외부와 통하는 약속. 헥사고날 용어 |
| 어댑터 (Adapter) | 포트의 실제 구현. Service나 Client가 어댑터 역할 |
| Mock | 테스트나 통합 전 단계에서 쓰는 가짜 구현체 |
| `client/{타도메인}/...UseCaseMock.java` | 호출자 도메인 안에 두는 임시 구현체 자리 |

## 6. Controller — `BibleController.java`

**파일 경로**: `qtai-server/src/main/java/com/qtai/domain/bible/web/BibleController.java`

외부 HTTP 요청이 가장 먼저 도착하는 자리. 김지민이 `NoteController`, `SharingController` 를 작성할 때 기반이 되는 패턴.

### 6-1. `web` 패키지 위치

```java
package com.qtai.domain.bible.web;
```

- `api`, `internal`, `client` 가 아닌 `web` — HTTP Controller와 외부 HTTP Request/Response DTO 자리.
- 다른 도메인이 `web` 패키지를 import할 일은 거의 없음 (외부 진입점이라서).

### 6-2. `@RestController` + `@RequestMapping`

```java
@RestController
@RequestMapping("/api/v1/bible")
public class BibleController { ... }
```

- `@RestController` = `@Controller` + `@ResponseBody`. **메서드 반환값을 자동으로 JSON으로 변환해서 응답.**
- `@RequestMapping("/api/v1/bible")` 은 모든 엔드포인트가 공유하는 base path. 메서드에서 `@GetMapping("/books")` 만 쓰면 실제 경로는 `/api/v1/bible/books` 가 된다.

> 적용:
> - `NoteController` → `@RequestMapping("/api/v1/notes")`
> - `SharingController` → `@RequestMapping("/api/v1/sharing-posts")`

### 6-3. Controller는 UseCase에 의존

```java
@RequiredArgsConstructor
public class BibleController {
    private final GetBibleVerseUseCase getBibleVerseUseCase;
    private final ListChaptersUseCase  listChaptersUseCase;
    private final SearchBibleUseCase   searchBibleUseCase;
}
```

핵심 규칙 2개:

1. **Controller는 Repository를 직접 호출하지 않는다** (CLAUDE.md §3). 항상 UseCase(또는 Service)를 거친다.
2. 자기 도메인 Service라도 **UseCase 인터페이스 타입으로** 받는다. 이러면:
   - Service 구현이 바뀌어도 Controller 코드 변경 없음
   - 테스트할 때 Mock UseCase를 꽂아서 단위 테스트 가능

### 6-4. HTTP 메서드 어노테이션

| HTTP 메서드 | 어노테이션 | 의미 |
| --- | --- | --- |
| GET | `@GetMapping` | 조회 (안전, 멱등) |
| POST | `@PostMapping` | 생성 또는 비RESTful 액션 |
| PUT | `@PutMapping` | 전체 교체 |
| PATCH | `@PatchMapping` | 부분 수정 |
| DELETE | `@DeleteMapping` | 삭제 |

김지민의 9개 API를 분류:

| API | 메서드 | 경로 |
| --- | --- | --- |
| 개인 노트 생성 | POST | `/api/v1/notes` |
| 나눔 공개 | POST | `/api/v1/notes/{id}/share` |
| 나눔 목록 | GET | `/api/v1/sharing-posts` |
| 나눔 상세 | GET | `/api/v1/sharing-posts/{postId}` |
| 좋아요 | POST | `/api/v1/sharing-posts/{postId}/like` |
| 좋아요 취소 | DELETE | `/api/v1/sharing-posts/{postId}/like` |
| 댓글 작성/조회 | POST | `/api/v1/sharing-posts/{postId}/comments` |
| 댓글 삭제 | DELETE | `/api/v1/comments/{commentId}` |
| 공유글 삭제 | DELETE | `/api/v1/sharing-posts/{postId}` |
| 신고 | POST | `/api/v1/reports` |

### 6-5. 파라미터 받는 4가지 방식

```java
@PostMapping("/{postId}/comments")
public ApiResponse<CommentResponse> createComment(
    @PathVariable  Long postId,                   // URL 경로의 {postId}
    @RequestBody  @Valid CommentCreateRequest req, // 요청 본문 JSON
    @AuthenticationPrincipal LoginUser user        // 인증된 사용자
) { ... }
```

| 어노테이션 | 받는 것 | 예시 |
| --- | --- | --- |
| `@PathVariable` | URL 경로 안의 변수 | `/notes/{id}` 의 `id` |
| `@RequestParam` | 쿼리 스트링 | `?page=0&size=20` |
| `@RequestBody` | HTTP 요청 본문 (JSON → 객체 자동 변환) | `POST` 요청의 body |
| `@AuthenticationPrincipal` | 인증된 사용자 정보 | 로그인된 회원 객체 |

`@Valid` 를 `@RequestBody` 와 같이 쓰면 Request DTO에 붙인 검증 어노테이션이 자동 실행된다.

### 6-6. 공통 응답 형식

```java
return ApiResponse.success(noteResponse);
```

CLAUDE.md 명시 공통 Envelope:

```json
{
  "success": true,
  "data": { ... },
  "error": null,
  "timestamp": "...",
  "traceId": "..."
}
```

이승욱이 만들 `ApiResponse.success(...)` 헬퍼를 호출하면 자동으로 위 형식이 된다. Controller에서 매번 JSON을 직접 조립하지 않는다.

### 6-7. 김지민의 `NoteController` 예상 모양

지금까지 학습한 어노테이션이 모두 등장한다.

```java
@RestController
@RequestMapping("/api/v1/notes")
@RequiredArgsConstructor
public class NoteController {

    private final CreateNoteUseCase createNoteUseCase;
    private final GetNoteUseCase    getNoteUseCase;
    private final UpdateNoteUseCase updateNoteUseCase;
    private final DeleteNoteUseCase deleteNoteUseCase;

    @PostMapping
    public ApiResponse<NoteResponse> create(
        @RequestBody @Valid NoteCreateRequest request,
        @AuthenticationPrincipal LoginUser user
    ) {
        NoteResponse created = createNoteUseCase.create(request, user.getId());
        return ApiResponse.success(created);
    }

    @GetMapping("/{id}")
    public ApiResponse<NoteResponse> get(@PathVariable Long id) {
        return ApiResponse.success(getNoteUseCase.getById(id));
    }

    @PutMapping("/{id}")
    public ApiResponse<NoteResponse> update(
        @PathVariable Long id,
        @RequestBody @Valid NoteUpdateRequest request,
        @AuthenticationPrincipal LoginUser user
    ) {
        return ApiResponse.success(updateNoteUseCase.update(id, request, user.getId()));
    }

    @DeleteMapping("/{id}")
    public ApiResponse<Void> delete(
        @PathVariable Long id,
        @AuthenticationPrincipal LoginUser user
    ) {
        deleteNoteUseCase.delete(id, user.getId());
        return ApiResponse.success(null);
    }
}
```

### 6-8. 핵심 용어 정리표 (6번 파일에서 등장)

| 용어 | 한 줄 요약 |
| --- | --- |
| `@RestController` | JSON 응답을 자동 처리하는 Controller |
| `@RequestMapping` | Controller의 base path 지정 |
| `@GetMapping`, `@PostMapping`, ... | HTTP 메서드별 엔드포인트 매핑 |
| `@PathVariable` | URL 경로 변수 수신 |
| `@RequestParam` | 쿼리 스트링 수신 |
| `@RequestBody` | 요청 JSON을 객체로 자동 변환 |
| `@Valid` | Request DTO 검증 어노테이션 활성화 |
| `@AuthenticationPrincipal` | 인증된 사용자 정보 주입 |
| `ApiResponse` (Envelope) | 모든 API의 공통 응답 형식 |
| 멱등성 (Idempotent) | 같은 요청을 여러 번 보내도 결과가 같음 (GET, PUT, DELETE) |
