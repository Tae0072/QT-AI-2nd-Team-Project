# 낙관적 업데이트(Optimistic Update) 학습 노트

> 작성일: 2026-06-12 · 계기: 나눔 좋아요(③ QA) 구현
> 관련 코드: `sharing_providers.dart`(`SharingFeedNotifier`/`SharingDetailNotifier`의 `toggleLike`)

---

## 1. 한 줄 정의

**서버 응답을 기다리지 않고, "성공할 것이라 낙관하고" UI를 먼저 바꾼 뒤, 서버에 요청을 보낸다. 실패하면 되돌린다(롤백).**

- 낙관적(Optimistic) = "어차피 성공할 거야"라고 낙관하고 화면부터 갱신.
- 반대말 = 비관적(Pessimistic) = 서버 응답을 받은 **뒤에야** 화면을 바꿈.

---

## 2. 왜 쓰나 — 우리가 겪은 문제

좋아요 하트를 누를 때 두 가지 방식이 있다.

### (A) 비관적 — 우리가 고치기 전 방식

```
하트 탭 → 서버에 like 요청 → 응답 옴 → 목록 전체 다시 조회(invalidate) → 화면 새로 그림
```

- 문제 1: 응답 올 때까지(수백 ms) 하트가 안 바뀌어 **반응이 굼떠 보임**.
- 문제 2: `invalidate`로 **전체를 다시 불러오니 화면이 깜빡임**(로딩 → 다시 데이터).
  - 상세(S-02)에선 좋아요 하나 눌렀는데 **본문·댓글까지 다 다시 로딩**됐다. 이게 김지민이 본 "재렌더".

### (B) 낙관적 — 고친 방식

```
하트 탭 → (즉시) 하트 채우고 +1 → 서버에 like 요청
                                   ├─ 성공: 그대로 둠 (이미 화면 맞음)
                                   └─ 실패: 하트 원복 -1 + "실패" 안내
```

- 누르자마자 반응 → **빠릿함**.
- 그 항목 하나만 바꾸니 **깜빡임 없음**(전체 재조회 안 함).

## 3. 핵심 3단계 (코드와 매칭)

`SharingFeedNotifier.toggleLike` 기준:

```dart
Future<void> toggleLike(int postId) async {
  final current = state.valueOrNull;
  if (current == null) return;
  final index = current.items.indexWhere((e) => e.id == postId);
  if (index < 0) return;
  final original = current.items[index];   // ← 롤백용 원본 보관
  final liked = original.likedByMe;

  // 1) 낙관적 갱신: 서버 요청 "전에" 화면부터 바꾼다
  state = AsyncData(_replace(current, index,
    original.copyWith(likedByMe: !liked,
                      likeCount: original.likeCount + (liked ? -1 : 1))));

  // 2) 서버 반영
  try {
    liked ? await repo.unlike(postId) : await repo.like(postId);
  } catch (_) {
    // 3) 실패 → 롤백: 보관해둔 원본으로 되돌린다
    final now = state.valueOrNull;
    if (now != null) {
      final i = now.items.indexWhere((e) => e.id == postId);
      if (i >= 0) state = AsyncData(_replace(now, i, original));
    }
    rethrow; // 화면이 "실패" 스낵바를 띄울 수 있게 다시 던짐
  }
}
```

| 단계 | 하는 일 | 빠뜨리면 |
| --- | --- | --- |
| ① 원본 보관 | 바꾸기 전 상태(`original`)를 변수에 저장 | 실패해도 되돌릴 게 없음 |
| ② 낙관적 갱신 | `await` **이전에** 동기로 `state` 변경 | 즉각 반응이 사라짐(=비관적) |
| ③ 실패 롤백 | `catch`에서 원본 복원 + 사용자 안내 | 화면과 서버가 어긋난 채 남음(거짓 좋아요) |

> 포인트: **낙관적 갱신은 반드시 `await`보다 먼저**, 동기로 일어나야 한다. 테스트도 이걸 검증한다
> (`toggleLike` 호출 직후, `await` 하기 전에 이미 +1 돼 있는지).

---

## 4. 왜 `FutureProvider`가 아니라 `AsyncNotifier`였나

- `FutureProvider`는 "한 번 불러오는 결과"만 들고 있어서, **부분 수정 메서드를 붙일 수 없다**.
  바꾸려면 `invalidate`(=전체 다시 불러오기)뿐 → 깜빡임의 원인.
- `AsyncNotifier`는 `state =`로 **들고 있는 값을 직접 부분 수정**할 수 있다. 그래서 `toggleLike` 같은
  메서드를 둘 수 있고, 해당 항목만 바꾼다.
- 상세는 `family`라서 `AsyncNotifierProvider.autoDispose.family`(postId별 인스턴스)를 썼다.

---

## 5. 상태 동기화에서 배운 점

- 상세에서 좋아요 → 상세는 낙관적으로 바꿨지만, **뒤에 있는 피드 목록**의 좋아요 수는 그대로다.
  → 상세 좋아요 성공 후 **피드는 `invalidate`로 백그라운드 동기화**. 사용자는 상세를 보고 있어서
    피드가 다시 불려도 **깜빡임이 안 보인다**. 돌아가면 수가 맞아 있다.
- 즉, "사용자가 보고 있는 화면 = 낙관적(안 깜빡임)", "안 보이는 화면 = 그냥 invalidate" 로 나눴다.

---

## 6. 주의/한계 (실수 방지)

- **롤백 안 하면 거짓 UI**: 서버는 실패했는데 화면엔 좋아요가 남는다 → 반드시 `catch` 롤백.
- **연타**: 빠르게 두 번 누르면 like→unlike가 연달아 나간다. 현재는 매 탭마다 현재 상태 기준으로
  토글하므로 최종 상태는 일관되지만, 서버 호출 순서 경합은 v1 허용 범위로 둠.
- **낙관적 갱신은 화면 유지 동안만**(autoDispose). 화면을 떠났다 다시 오면 서버 기준으로 재조회된다.
- **모든 곳에 쓰면 안 됨**: 결제처럼 실패 비용이 큰 동작은 낙관적이 부적절. 좋아요·북마크처럼
  "실패해도 가볍게 되돌릴 수 있는" 동작에 적합.

---

## 7. 한 문장 요약

> 좋아요 같은 가벼운 동작은 **서버를 기다리지 말고 화면부터 바꾸고(낙관), 실패하면 되돌린다(롤백)**.
> 그래서 빠릿하고 안 깜빡인다. 단, **원본 보관 + 실패 롤백**을 빠뜨리면 화면과 서버가 어긋난다.
