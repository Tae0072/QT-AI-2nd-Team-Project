## 0. 큰 그림 — 화면이 데이터를 받는 4단 구조 (식당 비유)

```
[screen]  →  [provider]  →  [repository(service)]  →  서버
 손님(주문)    웨이터          주방                      식자재 창고
```

| 폴더 | 역할 | 비유 | 오늘 만든 파일 |
|---|---|---|---|
| `models/` | 서버 JSON ↔ Dart 객체 틀 | 메뉴판/그릇 | `note_models.dart` |
| `services/` | 서버 호출(Dio) | 주방 | `note_repository.dart` |
| `providers/` | 상태 보관·화면에 공급(Riverpod) | 웨이터 | `note_providers.dart` |
| `screens/` | UI 그리기 | 손님 테이블 | `note_list_screen.dart` 등 |

핵심: **화면은 "데이터를 어디서 어떻게 가져오는지" 몰라도 된다(관심사 분리).** 출처가 바뀌어도 repository만 고치면 됨.

## 1. 모델 (데이터 클래스)

### `fromJson` 팩토리 생성자
서버가 준 JSON(`Map`)을 Dart 객체로 바꾸는 변환기.
```dart
factory NoteListItem.fromJson(Map<String, dynamic> json) {
  return NoteListItem(
    id: json['id'] as int,
    title: json['title'] as String? ?? '',   // null 안전 기본값
  );
}
```
- **왜 모델로 바꾸나**: ① 오타·타입 실수를 컴파일 때 잡음(`item.titel` → 빨간 줄) ② 파싱을 한 곳에 모음(응답 바뀌면 모델 1개만 수정).
- **null 안전 기본값** `as String? ?? ''`: 서버 값이 없거나 null이어도 앱이 안 죽게 기본값을 줌.

### 페이징 응답 (`content` / `last`)
백엔드 목록은 `content[]`(데이터) + `last`(마지막 페이지냐?)로 옴.
```dart
hasNext: !(json['last'] as bool? ?? true),  // last를 뒤집어 "다음 있나?"로 보관
```
- `last == true`(마지막) → `hasNext == false`. 화면에서 `if (hasNext) 더불러오기()`로 읽기 쉬우라고 뒤집음.

## 2. Riverpod (상태 관리)

| 종류 | 용도 | 오늘 예시 |
|---|---|---|
| `Provider` | 안 바뀌는 객체 공급 | `noteRepositoryProvider` (repository 주입) |
| `StateProvider` | 바뀌는 단순 상태 1개 | `noteCategoryFilterProvider` (선택된 카테고리) |
| `FutureProvider.autoDispose` | 비동기 데이터(서버 조회) | `notesProvider` (노트 목록) |

### `ref.watch` / `ref.read` / `ref.invalidate`
- `ref.watch(x)` — x를 **구독**. x가 바뀌면 화면 다시 그림. (build 안에서)
- `ref.read(x)` — 한 번만 **읽기**. 버튼 콜백 등에서. (구독 X)
- `ref.invalidate(x)` — x를 **무효화** → 다음에 다시 조회. (저장 후 목록 새로고침에 사용)

### 자동 재조회 패턴
```dart
final notesProvider = FutureProvider.autoDispose((ref) {
  final category = ref.watch(noteCategoryFilterProvider); // 필터를 watch
  return repository.getNotes(category: category);
});
```
→ 카테고리 칩을 바꾸면 `notesProvider`가 **자동으로 다시 조회**됨. `autoDispose` = 화면 떠나면 캐시 정리.

### `ConsumerWidget` vs `ConsumerStatefulWidget`
- `ConsumerWidget` — Riverpod 쓰는 무상태 화면. (N-01 목록)
- `ConsumerStatefulWidget` — Riverpod + 자체 상태(컨트롤러·플래그) 필요할 때. (N-03 작성: TextEditingController, `_saving`)

## 5. 속성(파라미터) 포인트

### `BottomNavigationBar`의 `type`
```dart
BottomNavigationBar(type: BottomNavigationBarType.fixed, ...)
```
- 탭 **4개 이상**이면 기본값이 `shifting`(선택 안 된 라벨 숨김)으로 바뀜 → `fixed`를 직접 줘서 **라벨 항상 표시**.

### `TextField` 여러 줄 입력
```dart
TextField(maxLines: null, expands: true, ...) // 남는 높이만큼 늘어나는 본문칸
```

### `TextEditingController`
- 입력값을 읽고/지우는 컨트롤러. **반드시 `dispose()`** 해서 메모리 누수 방지.
```dart
final _bodyController = TextEditingController();
@override void dispose() { _bodyController.dispose(); super.dispose(); }
```

### `mounted` 체크
- `await` 뒤에 위젯이 화면에서 사라졌을 수 있어, `context` 쓰기 전에 `if (!mounted) return;`로 안전 확인.

## 6. 화면 이동(라우팅)

### 라우트 이름 상수 (`static const String`)
```dart
class AppRouter {
  static const String noteEdit = '/notes/edit';
}
```
- "마법의 문자열"을 한 곳에 모음 → 오타를 컴파일 때 잡고, 주소 바꿀 때 한 곳만 수정, 자동완성.

### 이동 3단계
```
1) Navigator.pushNamed(context, AppRouter.noteEdit, arguments: 'PRAYER')
2) onGenerateRoute(settings)  // settings.name == '/notes/edit'
3) switch에서 같은 이름 case → MaterialPageRoute(builder: 화면 위젯) 반환 → 등장
```
- **`MaterialPageRoute`** = 화면을 슬라이드 애니메이션으로 감싸 Navigator에 넘기는 "포장 상자". 위젯이 아니라 화면 전환을 담는 클래스.
- **arguments** = 이동에 값을 실어 보냄(택배). 받는 쪽: `ModalRoute.of(context)!.settings.arguments`.
  - ⚠️ `onGenerateRoute`에서 `MaterialPageRoute(settings: settings, ...)`로 **settings를 넘겨야** arguments가 전달됨.
- **provider vs arguments**: 일회성 값 전달 = arguments(잔여물 없음), 앱 전체 공유 상태 = provider.

### `Navigator.popUntil`
```dart
Navigator.of(context).popUntil(
  (route) => route.settings.name == AppRouter.noteList || route.isFirst,
);
```
- 저장 후 N-03 → (N-02 건너뛰고) **목록까지 한 번에 닫고 돌아가기**.

## 7. const / static

- `const` — 안 변하는 컴파일 타임 상수. 위젯 앞 `const`는 재생성 안 해 성능에도 도움.
- `static` — 객체(인스턴스) 없이 `클래스.멤버`로 접근하는 공용 값.
- ⚠️ `const [ ... ]` 리스트 안에는 **const 위젯만** 들어갈 수 있음(에러로 배움).
- ⚠️ 파일 맨 위 설명은 `//`(일반 주석). `///`(문서 주석)은 바로 아래 선언에 붙어야 함 — 떠 있으면 "dangling doc comment" 경고.

## 9. 실행·검증 메모

- `flutter analyze` — 정적 분석(빨간 줄·경고). 화면 만들 때마다 돌려 무경고 유지.
- `flutter run -d emulator-5554 --dart-define=DEV_FORCE_HOME=true`
  - `--dart-define=DEV_FORCE_HOME=true` = 로그인·온보딩 **건너뛰고 바로 홈**(개발용 스위치).
  - 단, 로그인 안 하므로 **목록·저장 API는 401**(에러 안내가 정상 동작). 실데이터는 백엔드 + 카카오 로그인(레벨2) 필요.
- `flutter run` 실패해도 `√ Built` + `Installing`이 떴으면 **앱은 설치·실행됨**. exit 2가 `service protocol` 연결(핫리로드)만 실패한 경우 앱 자체는 정상.

## 한 줄 요약

화면(screen)은 provider를 watch만 하고, provider는 repository를, repository는 Dio로 서버를 호출한다. JSON은 fromJson으로 모델로 바꾸고, 화면 이동은 라우트 이름 상수 + onGenerateRoute의 case + MaterialPageRoute로 연결한다.
