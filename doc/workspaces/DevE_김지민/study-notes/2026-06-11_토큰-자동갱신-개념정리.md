# 토큰 자동 갱신(refresh) 개념 정리 — admin-web P2

> 작성: 2026-06-11 / 맥락: admin-web 코드리뷰 P2(토큰 자동 갱신) 구현하며 정리.
> 한 줄 요약: **access는 짧게 자주 쓰고, 만료되면 refresh로 조용히 새 access를 받아 끊김 없이 쓴다. refresh까지 죽으면 그때만 다시 로그인.**

---

## 1. 토큰이 왜 2개인가

| 토큰 | 수명 | 역할 | 어디에 |
| --- | --- | --- | --- |
| **access** | 30분(짧음) | **매 API 요청**에 `Authorization: Bearer` 로 붙여 신분 증명 | 매 요청 헤더 |
| **refresh** | 14일(김) | access가 만료됐을 때 **"새 access 주세요"** 하고 내미는 열쇠 | refresh 요청 body |

- access를 짧게 두는 이유: 탈취당해도 30분이면 무효 → 보안.
- 매번 로그인하긴 번거로우니, 긴 수명의 refresh로 access만 조용히 재발급.

## 2. 자동 갱신 흐름 (핵심)

```
어떤 API 요청 → access 만료라 서버가 401
  → (인터셉터가 가로챔) refresh 토큰으로 POST /api/v1/auth/refresh
  → 새 access 받음 → 저장 → 방금 실패한 그 요청을 새 access로 1번 더
  → 성공 → 사용자는 아무것도 모름 (화면 그대로, 끊김 없음)
```

- 사용자는 "로그인 풀렸다" 경험 없이 계속 사용 = UX 좋아짐.
- 핵심은 **"401 → 갱신 → 원요청 재시도"** 를 axios **응답 인터셉터**가 자동으로 해준다는 것.

## 3. "refresh가 망한다" = 그때만 다시 로그인

refresh 토큰을 내밀었는데도 새 access를 **못 받는** 경우:
- refresh도 만료(14일 지남) / 무효(세션 끊김) / 회원 탈퇴(M0006)·정지(M0007)

→ 더는 자동 갱신 불가 = **진짜 재로그인 필요** → 로그인 화면으로.

| 상황 | 처리 | 화면 |
| --- | --- | --- |
| access만 만료(흔함) | refresh 성공 → 자동 갱신 | 그대로 유지 |
| refresh까지 실패(드묾) | 세션 종료 | 로그인 화면 |

## 4. 헷갈리기 쉬운 함정 2개

### ① single-flight (동시 401에 refresh는 1번만)

화면이 한 번에 여러 API를 부르면 401이 **동시에 여러 개** 뜬다. 각자 refresh하면 같은 갱신이 5번, 10번 중복 발생.
→ **공유 Promise 1개**를 두고, 이미 갱신 중이면 다들 그 Promise를 같이 기다린다. 갱신은 딱 1번.

```ts
let refreshing: Promise<string> | null = null;
function refreshOnce() {
  refreshing ??= 실제갱신().finally(() => { refreshing = null; }); // 끝나면 슬롯 비움
  return refreshing;
}
```

### ② 인터셉터는 React가 아니다 → "콜백"으로 화면 전환 (B 방식)

`client.ts`(axios 인터셉터)는 **React 컴포넌트가 아니라 그냥 함수**라 `useNavigate`/`useAuth`를 **못 쓴다**. 그래서 "로그인 화면으로 보내"를 어떻게 전달하냐가 문제.

- **콜백 슬롯(전화번호) 방식 = B**:
  1. `client.ts`가 빈 슬롯 + 등록함수(`setAuthExpiredHandler`)를 둠.
  2. `AuthContext`가 앱 켜질 때 자기 정리함수(`clearSession`)를 등록.
  3. refresh 최종 실패 → client가 그 콜백 호출 → 토큰 비움 → `ProtectedRoute`가 알아서 `/login`. (새로고침 없음)
- A 방식(`window.location='/login'` 하드 리로드)은 간단하지만 앱이 통째 리셋 → 이 프로젝트의 SPA 흐름과 안 맞아 B 채택.

## 5. 코드 어디에 뭐가 있나 (admin-web)

| 파일 | 역할 |
| --- | --- |
| `auth/tokenStorage.ts` | access·refresh 저장/조회, `clearToken`은 둘 다 제거 |
| `api/client.ts` | 401→single-flight refresh→원요청 재시도, 실패 시 `endSession()`(콜백) |
| `auth/AuthContext.tsx` | `login(access, refresh?)` 저장, `setAuthExpiredHandler(clearSession)` 등록 |
| `pages/LoginPage.tsx` | 카카오 로그인 시 `login(accessToken, refreshToken)` |
| `vite.config.ts` | dev 프록시 `/api/v1/auth`→service-user(8081) (refresh는 JWT 발급자 소관) |

## 6. 서버 계약 (이승욱)

- 공용 `POST /api/v1/auth/refresh` 재사용(관리자 전용 경로 없음). 재발급 access의 `role=ADMIN`은 서버가 DB 다시 읽어 유지.
- access 30분 / refresh 14일. M0006(탈퇴)/M0007(정지)면 갱신 거부 → 로그인 화면.
