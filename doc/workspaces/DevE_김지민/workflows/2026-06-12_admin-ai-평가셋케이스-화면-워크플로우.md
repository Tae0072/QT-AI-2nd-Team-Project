## 배경 / 결정

- 서버 `AdminAiEvaluationController`는 **완전 구현**돼 있는데 admin-web에 화면이 없었다(순수 FE 작업).
- QA 기간 역할 분담(파일/화면 단위 계약): **나 = 평가셋/케이스**(신규 `aiEvaluations.ts`+`AiEvaluationsPage.tsx`,
  `createEvaluationCandidate` 소유, 공용파일 먼저 머지) / **팀원 = AI Asset 상세·재생성**(`AiAssetsPage`+`aiAssets.ts`).
  → **나는 `AiAssetsPage`/`aiAssets.ts`를 건드리지 않는다.**
- Req2(프롬프트 버전·sourceLabel **관리**)는 서버 수정 API 부재로 보류(읽기표시는 팀원 Asset 상세).

| 파일 | 변경 |

| `constants/menu.ts` | AD-11 메뉴(`/ai-evaluations`, `REVIEWER`+`CONTENT_CREATOR`) |
| `App.tsx` | `/ai-evaluations` 라우트 등록 |

- **권한 가드**: 페이지 접근 = REVIEWER/CONTENT_CREATOR/SUPER_ADMIN(RoleGuard). **케이스 승인/반려 버튼은
  REVIEWER/SUPER_ADMIN만**(`canAccessAdminRoute(role,[REVIEWER])`) — CONTENT_CREATOR는 생성까지만.
- **기존 컨벤션 정렬**: 기존 페이지가 쓰는 `destroyOnClose`로 통일(antd 경고 일괄 정리는 QA Workstream B).

## enum 검증(서버 대조)

| 항목 | 서버 enum | 값 |
| --- | --- | --- |
| evalType | `AiEvaluationType` | EXPLANATION / SIMULATOR / QA |
| targetType | `AiTargetType` | BIBLE_VERSE / QT_PASSAGE / QA_REQUEST |
| sourceType | `AiEvaluationSourceType` | VALIDATION_FAILURE / USER_REPORT / ADMIN_CREATED |
| set status | `AiEvaluationSetStatus` | DRAFT / ACTIVE / RETIRED |
| case status | `AiEvaluationCaseStatus` | CANDIDATE / APPROVED / REJECTED |

서버가 `parseEnum`으로 강제 검증 → 5개 셀렉트 옵션 전부 enum과 1:1 일치 확인.
