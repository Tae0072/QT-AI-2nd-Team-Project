## 배경 / 결정

- 서버 `AdminAiEvaluationController`는 **완전 구현**돼 있는데 admin-web에 화면이 없었다(순수 FE 작업).
- QA 기간 역할 분담(파일/화면 단위 계약): **나 = 평가셋/케이스**(신규 `aiEvaluations.ts`+`AiEvaluationsPage.tsx`,
  `createEvaluationCandidate` 소유, 공용파일 먼저 머지) / **팀원 = AI Asset 상세·재생성**(`AiAssetsPage`+`aiAssets.ts`).
  → **나는 `AiAssetsPage`/`aiAssets.ts`를 건드리지 않는다.**
- Req2(프롬프트 버전·sourceLabel **관리**)는 서버 수정 API 부재로 보류(읽기표시는 팀원 Asset 상세).

## enum 검증(서버 대조)

| 항목 | 서버 enum | 값 |
| --- | --- | --- |
| evalType | `AiEvaluationType` | EXPLANATION / SIMULATOR / QA |
| targetType | `AiTargetType` | BIBLE_VERSE / QT_PASSAGE / QA_REQUEST |
| sourceType | `AiEvaluationSourceType` | VALIDATION_FAILURE / USER_REPORT / ADMIN_CREATED |
| set status | `AiEvaluationSetStatus` | DRAFT / ACTIVE / RETIRED |
| case status | `AiEvaluationCaseStatus` | CANDIDATE / APPROVED / REJECTED |

서버가 `parseEnum`으로 강제 검증 → 5개 셀렉트 옵션 전부 enum과 1:1 일치 확인.

## 추가 작업 (2026-06-12) — dev 머지 + 후보 버튼 연결 + 라벨 정리

- **dev 머지**: 팀원 PR `#561`(AI Asset 상세/재생성)이 dev에 머지돼 `git merge origin/dev` 수령(충돌 없음 —
  팀원은 `AiAssetsPage`/`aiAssets.ts`만 건드림). Req2 읽기표시(sourceLabel·promptVersion)는 팀원 상세 화면에 포함 확인.
- **평가 항목으로 추가(후보 버튼) 연결** — 팀원이 다른 작업 중이라(QA 기간 다같이) 합의 하에 내가 직접 연결:
  - `AiAssetsPage` 상세 드로어 `extra`에 **"평가 항목으로 추가"** 버튼 + 모달 추가. `aiEvaluations.ts`의
    `createEvaluationCandidate` import.
  - 모달은 **평가 세트 드롭다운만**(대상유형 같은 세트만 `listEvaluationSets({targetType})`로 필터, 없으면 안내).
    JSON 입력 없음 — 서버가 asset에서 `inputJson` 자동 생성, `sourceType=VALIDATION_FAILURE` 자동.
  - 서버 제약 검증: `createAssetCandidate`의 유일 호환 조건은 `asset.targetType == set.targetType` → 드롭다운 필터로 원천 차단.
- **라벨 쉬운 말 정리(세트/항목)**: 메뉴 `AI 평가 세트`, 버튼 `평가 세트 만들기`/`항목 보기`/`평가 항목 추가`/
  `평가 항목으로 추가`. (사용자 노출 문자열 잔여 0, 내부 주석은 API 용어 유지.)
- **JSON 입력 정책(결정)**: 서버 계약상 세트의 `expectedPolicyJson`은 선택, 케이스의 `inputJson`은 필수 JSON.
  주 경로(후보 등록)는 JSON 없음. 직접 만들기 폼만 JSON 노출 → **현행 유지**(파워 경로). 구조화 폼은 evalType별
  형태가 달라 백엔드 합의 필요.
