## 1. 배경 / 문제
AD-01 대시보드의 해설 "없음" 상태에서, 현재 CTA는 임시로 "해설 검증하러 가기"(`/ai-assets`)뿐이다. AD-03은 **기존 자산의 승인/반려/숨김/재생성(검증)**만 가능하고, **미생성 해설을 처음부터 만드는 "생성 요청"이 없다**(생성=배치/시스템). 관리자가 특정 본문의 해설 생성을 직접 트리거할 수 있게 한다.

## 2. 핵심 재사용
백엔드에 이미 `AiDailyQtVerseExplanationSeedService.seedToday()`(00:05 배치)가 있다 — QT 본문 절들에 EXPLANATION 생성 job을 만들고, 이미 승인됐거나 진행 중인 절은 스킵한다. 이를 **본문 단위(`seedForPassage`)로 일반화**해 재사용한다. 컨텍스트는 `GetQtPassageContentContextUseCase.getContentContext(qtPassageId)`로 조회.

## 3. MSA 동기화 범위 (중요)
`AiService`·`AiDailyQtVerseExplanationSeedService`는 **admin-server와 service-ai 양쪽에 존재**한다(CLAUDE.md 동기화 규칙).
- **도메인 로직**(`seedForPassage`, AiService 오케스트레이션) → **service-ai(원본) 먼저 → admin-server 동기화.**
- **admin 컨트롤러**(트리거 엔드포인트) → admin-server 단독(service-ai의 `/api/v1/admin/**`은 denyAll 봉인).

## 4. 작업 단계

### 백엔드 (service-ai 원본 → admin-server 동기화)
1. [ ] `AiDailyQtVerseExplanationSeedService`: 핵심 시딩을 `seedForContext(context, requestedBy)`로 추출, `public seedForPassage(qtPassageId, requestedBy)` 추가(`getContentContext` 사용, 본문 없음/빈 절 처리). `seedToday()`는 `seedForContext(..., SYSTEM_BATCH)`로 위임.
2. [ ] api: `GenerateQtPassageExplanationUseCase`(interface) + `GenerateExplanationResult`(created/skipped/failed/reason) DTO.
3. [ ] `AiService`: 위 UseCase 구현 — seedForPassage 호출 + 감사로그 `AI_EXPLANATION_GENERATE_REQUEST`(adminId/actorType) 기록 + api 결과 매핑.
4. [ ] admin 컨트롤러(admin-server 단독): `POST /api/v1/admin/ai/qt-passages/{qtPassageId}/explanations/generate` (권한 REVIEWER/SUPER_ADMIN = `requireReviewer`), 결과 반환.
5. [ ] `AuditQueryService` AI action allowlist에 `AI_EXPLANATION_GENERATE_REQUEST` 추가.
6. [ ] 위 1~3·5를 service-ai ↔ admin-server **동일 동기화**.

### 프런트 (admin-web)
7. [ ] api: `generateQtPassageExplanation(qtPassageId)`.
8. [ ] `DashboardPage`: 해설 "없음" CTA를 "해설 검증하러 가기"(navigate) + **"해설 생성 요청"(POST, 확인 후 호출)** 으로. 성공 시 "생성 요청됨 — 잠시 후 검증 대기에 표시" 안내.

### 테스트 (CLAUDE.md §10)
9. [ ] service 단위: seedForPassage(생성/스킵/본문없음), AiService 감사로그 기록.
10. [ ] 컨트롤러 보안/권한 테스트(REVIEWER 허용, USER 차단).
