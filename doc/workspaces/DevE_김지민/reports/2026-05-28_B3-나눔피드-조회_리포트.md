### 1. Entity ↔ 명세 갭 발견 + 이승욱 협의 (A안)

`sharing_posts` Entity에 07 §F-10이 명시한 `nickname_snapshot` 컬럼이 누락돼 있었다. 닉네임 나눔은 작성자명에 닉네임을 박제해야 하므로(닉네임 변경·노트 수정에 영향 없게) 컬럼이 필요했다.

- **이승욱 협의 완료**: nickname_snapshot + verse 라벨 컬럼 추가에 합의
- **V13 마이그레이션**으로 ALTER TABLE (V5는 이미 적용돼 수정 불가 → 새 버전으로 추가)
  - `nickname_snapshot VARCHAR(20) NULL` — 작성자 닉네임 박제 (07 §F-10)
  - `snapshot_verse_label VARCHAR(100) NULL` — 본문 범위 라벨 (예: "창세기 1:1-5", 04 §4.4.1)
- nullable 선택 이유: 발행 로직(`PublishNoteUseCase`)이 아직 미구현이라 채울 코드가 없음. DB는 열어두고 발행 로직에서 반드시 채우는 책임을 지운다. 발행 흐름 완성 후 NOT NULL 조이기는 후속.

### 2. B3 구현

| 계층 | 파일 | 역할 |
|---|---|---|
| web | `SharingPostController` | `GET /api/v1/sharing-posts`, 인증·파라미터 추출 |
| api | `ListSharingPostsUseCase` | 포트(계약) |

| internal | `SharingPostService` | 필터·정렬·N+1방지·매핑 |
| internal | `SharingPostRepository` | PUBLISHED + category + q 검색 |
| internal | `PostLikeRepository` | likedByMe 배치 조회 |

구현 포인트:
- **응답 DTO 분리(A)**: 목록(`bodyPreview`)과 상세(B4, 전체 본문+절배열)는 필드가 달라 분리. 노트 때(NoteCreate/UpdateResponse)와 같은 결정.
- **likedByMe 배치 조회**: 글마다 조회하면 N+1(1+N). `post_likes`를 `IN` 절로 1회에 모아 Set으로 매핑 → 항상 쿼리 2번.
- **정렬 화이트리스트**: `publishedAt→createdAt`, `likeCount`, `commentCount`만 허용. 허용 외 필드는 무시하고 기본 정렬. 클라이언트가 임의 필드로 정렬하는 것을 차단(보안·안정성).
- **q LIKE 이스케이프**: 사용자 입력의 `%`, `_`, `\`를 리터럴로 이스케이프(노트 패턴 재사용).
- **status/category는 String**: `SharingPostStatus` enum이 internal에 있어 api/dto가 import하면 도메인 경계 위반 → String으로.

### 4. B4 나눔 상세 구현 (GET /api/v1/sharing-posts/{postId})

같은 브랜치(`feature/sharing-posts`)에서 이어 구현. 목록과 상세는 응답 모양이 달라 DTO를 분리.

| 계층 | 추가 |
|---|---|
| api | `GetSharingPostUseCase` 포트 |
| api/dto | `SharingPostResponse`(상세, §4.4.2) + `VerseSnapshotDetail`(rangeLabel+verses[]) + `VerseLine` |
| internal | `SharingPostRepository.findByIdAndStatus()` (PUBLISHED만) |
| internal | `SharingPostService.getDetail()` — 404 + likedByMe + ownedByMe |
| web | `SharingPostController` GET `/{postId}` |

구현 포인트:
- **404 처리**: `findByIdAndStatus(id, PUBLISHED)` → `orElseThrow(SHARING_POST_NOT_FOUND)`. HIDDEN/DELETED/없는 글은 애초에 안 잡혀 자동 404(존재를 숨김, §4.4.2).
- **likedByMe**: 단건이라 `findLikedPostIds(memberId, List.of(postId))` 비었는지로 판단(목록 배치 메서드 재사용).
- **ownedByMe**: `post.memberId == 조회자`.
- 테스트 +6건(서비스 3 + 컨트롤러 2 + 통합 1) → **나눔 조회 전체 17건**.

### ⚠️ verses[] (다중 절 본문) = v2 이관

§4.4.2 상세 응답의 `verseSnapshot.verses[]`(절별 본문 배열)는 **빈 배열로 반환**한다. 이유:
- 나눔에 **개별 절 본문을 저장하는 수단이 전혀 없음**(확인 완료: `sharing_post_verses` 테이블/엔티티 없음, V5/V13에도 절 본문 컬럼 없음). 우리가 가진 건 `snapshot_verse_label`(rangeLabel 한 줄)뿐.
- 07 §19.2: **다중 절 선택은 v2 기능**(v1은 단일 절만). 절 선택 방식 설정은 이승욱(마이페이지), 절 데이터는 이지윤(설교/note_verses) 소유.
- → verses[] **채우기는 김지민 단독 B4 작업이 아니라 v2 협의 작업**. B4는 명세 모양만 맞춰 필드 선언(빈 배열), 데이터는 v2에서.
- **v2 필요 작업**: `sharing_post_verses` 스냅샷 저장 구조 + 발행 로직이 note_verses를 복제 + 다중 절 선택 UI(이승욱 설정). 절 본문은 발행 시점 박제(스냅샷 정책).

|---|---|---|
| 1 | N+1 문제 | 목록 1번 + 항목마다 N번 조회 = 느림. 반복문 안 조회가 원인. `IN` 절 배치로 2번에 끝냄 |
| 2 | 정렬 화이트리스트 | sort는 URL 문자열이라 누구나 아무 필드나 넣음. 허용 목록만 통과시켜 에러·정보노출·성능공격 차단 |
| 3 | Flyway 자동 적용 | 마이그레이션은 앱 시작 시 Flyway가 자동 적용(사람 X). 적용 여부는 flyway_schema_history가 추적. 충돌 위험은 "버전 번호 중복"이라 만드는 사람을 한 명으로 협의 |
| 4 | 마이그레이션 불변 | 이미 적용된 V5는 수정 금지. 컬럼 추가는 새 버전(V13)으로 ALTER |
| 5 | BaseEntity 상속 | id/createdAt/updatedAt/deletedAt 공통 필드를 한 곳에 모아 중복 제거 + JPA Auditing 자동 시각. PostLike는 updatedAt 불필요해 상속 안 함 |
| 6 | 스냅샷 정책 | nicknameSnapshot은 발행 시점 박제. 닉네임 변경·노트 수정에도 안 바뀜(07 §F-10) |
