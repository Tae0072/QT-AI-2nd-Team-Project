## 1. 증상

- AD-01 대시보드: "AI 검증 대기 **857**" 정상 표시
- AD-03 AI 산출물 검증 목록: **아무것도 안 뜸**
- 그러나 `GET /admin/ai/assets?status=VALIDATING`는 데이터가 분명히 존재(857건)

## 2. 진단 (프론트/백엔드 분리)

### 2-1. 백엔드 API 직접 호출로 격리

| 호출 | 결과 |
|---|---|
| `status=VALIDATING` (857건) | 초기엔 정상, 이후 타임아웃 |
| status 없음 (전체 31,104건) | **30초 타임아웃 (HTTP 000)** |
| `status=APPROVED` (30,247건) | **타임아웃** |

→ 필터 없이/대량 조회 시 백엔드 쿼리가 응답을 못 함. PR1 이전 FE(기본필터 없음 → 전체조회)가 타임아웃 → "안뜸"의 직접 원인.

### 2-2. 실제 SQL과 코드

`AdminAiAssetQueryRepository`의 `LATEST_VALIDATION_JOIN`:
```sql
left join ai_validation_logs avl1_0 on avl1_0.id=(
  select max(avl2_0.id) from ai_validation_logs avl2_0
  where avl2_0.ai_asset_id=aga1_0.id
  and avl2_0.created_at=(
    select max(avl3_0.created_at) from ai_validation_logs avl3_0
    where avl3_0.ai_asset_id=aga1_0.id))
```
자산마다 "최신 검증로그"를 **이중 중첩 상관 서브쿼리**로 계산.

### 2-3. 근본 원인 (2가지)

| # | 원인 | 상세 |
|---|---|---|
| **1차** | `ai_validation_logs(ai_asset_id)` **인덱스 부재** | 정식 스키마 `V6__create_auth_ai_explanation_tables.sql`는 `fk_ai_validation_logs_asset` FK(→ InnoDB 자동 인덱스)와 `idx_ai_assets_target_status`·`idx_ai_assets_job`를 정의. 그러나 **이 로컬 DB는 정식 Flyway가 아니라 인덱스/FK 없는 불완전 스키마로 생성됨**(`flyway_schema_history` 없음). 인덱스가 없어 상관 서브쿼리가 자산 행마다 62k행 풀스캔 → 폭발 |
| **2차** | count 쿼리의 불필요한 검증 조인 | `count(asset.id)`만 세면 되는데 `LIST_FROM`(무거운 `LATEST_VALIDATION_JOIN` 포함)을 그대로 사용. `checklistVersionId` 필터가 없을 때도 항상 조인 |

> 추가로, 타임아웃된 클라이언트가 끊겨도 MySQL 서버 쿼리는 계속 실행되어 같은 목록 쿼리가 10건+ 동시 누적(최장 615초) → 커넥션/DB 포화로 빠른 VALIDATING 쿼리마저 막힘.

## 3. 수정

### 3-1. (1차, 핵심) 정식 V6 인덱스 복원 — 로컬 DB only

정식 스키마가 이미 정의한 인덱스를 이 DB에 복원(코드/마이그레이션 변경 아님, git 무관):
```sql
CREATE INDEX fk_ai_validation_logs_asset  ON ai_validation_logs (ai_asset_id);
CREATE INDEX idx_ai_assets_target_status  ON ai_generated_assets (target_type, target_id, status);
CREATE INDEX idx_ai_assets_job            ON ai_generated_assets (generation_job_id);
```

### 3-2. (2차) count 쿼리 코드 개선

**파일**: `qtai-server/admin-server/.../ai/internal/AdminAiAssetQueryRepository.java`

- `BASE_FROM`(검증 조인 없음) / `LIST_FROM`(= BASE_FROM + LATEST_VALIDATION_JOIN) 분리
- `BASE_WHERE`(latestValidation 미참조) / `CHECKLIST_WHERE`(latestValidation 참조) 분리
- **count 쿼리는 `checklistVersionId` 필터가 있을 때만** `LATEST_VALIDATION_JOIN` 포함. 없으면 검증 조인 없이 count
- 목록(select) 쿼리는 `latestValidation.result` 컬럼이 필요하므로 변경 없음

## 5. 영향 / 주의

- **인덱스 복원은 로컬 DB 런타임만** 변경. 정식보다 부족했던 걸 정식 수준으로 되돌린 것이라 프로젝트(레포·타 팀원)에 영향 없음.
- count 코드 수정은 결과값 불변, 불필요 연산만 제거.
- 별도 기록: import 시 `ai_generation_jobs`·`ai_generated_assets`에 nullable `updated_at` 컬럼을 추가했음(bundle이 구 스키마 기준). 정식으론 bundle을 현 스키마로 재생성하는 게 맞음.
- **권장**: 이 DB는 인덱스/FK 누락 상태이므로, 가능하면 정식 Flyway로 재구축하거나 누락 인덱스를 전수 점검할 것.
