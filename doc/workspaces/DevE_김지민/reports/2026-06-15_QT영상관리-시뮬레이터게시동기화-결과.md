## 1. 작업 범위

이번 작업은 두 흐름을 정리했다.

1. AI 산출물 `SIMULATOR + QT_PASSAGE` 승인본을 기존 해설/용어 승인 흐름과 대칭으로 study 게시 흐름에 연결했다.
2. 앱 재생용 QT 영상을 관리자웹에서 운영할 수 있도록 원본 영상, 절별 구간, QT 클립 관리 API와 화면을 추가했다.

## 2. 백엔드 변경

### 시뮬레이터 승인/숨김 동기화

- `AiAssetReviewService`
  - `APPROVE + activateForTarget=true`일 때 SIMULATOR payload를 읽어 `PublishApprovedSimulatorClipCommand`로 변환한다.
  - payload 계약은 `title`, `componentLibraryVersionId`, `sceneScript` 또는 `sceneScriptJson`이다.
  - `HIDE` 시 `HidePublishedSimulatorClipUseCase`를 호출해 공개 클립을 숨긴다.
- `AiAssetReviewServiceTest`
  - 승인 시 게시 호출
  - 승인만 하고 활성화하지 않는 경우 게시 미호출
  - 숨김 시 공개 클립 숨김 호출을 검증했다.

### QT 영상 관리 API

- 신규 `domain.qtvideo`
  - `source_videos`: 원본 영상 URL, duration, 저장소 유형, 상태 관리
  - `bible_verse_video_segments`: 성경 절별 시작/종료 초 구간 관리
  - `qt_video_clips`: QT 본문별 클립 생성/상태 관리
- `AdminQtVideoController`
  - `GET /api/v1/admin/qt-videos/bible-books`
  - `GET/POST/PATCH /api/v1/admin/qt-videos/source-videos`
  - `GET/PUT /api/v1/admin/qt-videos/source-videos/{sourceVideoId}/segments`
  - `GET /api/v1/admin/qt-videos/clips`
  - `POST /api/v1/admin/qt-videos/qt-passages/{qtPassageId}/clips/prepare`
  - `PATCH /api/v1/admin/qt-videos/clips/{clipId}/status`
- 권한은 `OPERATOR`, `REVIEWER`, `CONTENT_CREATOR`, `SUPER_ADMIN`을 허용한다.
- 감사 조회 allowlist에 QT 영상 액션/대상 유형을 추가했다.

## 3. 관리자웹 변경

- `QtVideosPage`
  - 원본 영상 등록/수정/조회
  - 성경권 드롭다운 조회
  - 절별 segment JSON 등록/수정
  - QT 본문별 클립 생성
  - 클립 상태 `APPROVED/HIDDEN/FAILED` 변경
- 메뉴에 `AD-20 QT 영상 관리`를 추가했다.
- 대시보드 시뮬레이터/QT 영상 CTA를 `/qt-videos?qtPassageId=`로 연결했다.

## 4. 로컬 확인 사항

- Docker MySQL 기준 성경 메타데이터 확인
  - `bible_books`: 66건
  - `bible_verses`: 31,104건
  - 고린도전서: `id=46`, `code=1CO`
  - 한글명은 터미널에서 `?????`처럼 보였지만 `HEX(korean_name)` 값이 UTF-8로 정상 저장되어 있어 콘솔 표시 문제로 판단했다.
- `http://localhost:8090/api/v1/admin/qt-videos/bible-books`
  - 비로그인 직접 호출은 `401`이다.
  - 이는 API 미존재(`404`)가 아니라 인증 필요 응답이다.

## 6. 남은 확인 포인트

- 브라우저에서 성경권 드롭다운 404가 보였던 건 DB 데이터 부재가 원인이 아니었다.
- 마지막 확인 시점에는 8090 포트를 `qtai-admin-server` Docker 컨테이너가 잡고 있었다. 로컬 `bootRun` 검증을 하려면 Docker admin-server를 내리거나 Vite proxy target을 실제 로컬 서버 포트로 맞춰야 한다.
- 로그인 상태에서 브라우저 네트워크 탭 기준 실제 404 URL을 다시 잡아야 한다. 현재 서버 직접 호출은 401로, 라우트 자체는 존재하는 것으로 확인됐다.
