## 1. 작업 요약

2026-06-15 리포트에서 “골격 구현/인계” 상태였던 앱 재생용 QT 영상 관리(AD-20)를 이어서 디버깅하고 운영 기능을 보강했다.

- 원본 영상 수정 400 오류를 수정했다.
- 절별 구간 모달에서 기본 예시 2건을 실데이터처럼 저장하는 실수를 막았다.
- 원본 영상 삭제와 QT 클립 삭제 기능을 추가했다.
- 원본 영상 삭제 시 해당 원본으로 만든 QT 클립과 절별 구간을 함께 삭제한다.
- 클립 목록에서 `클립 ID`, `QT 본문(날짜)`, `원본 영상 ID`를 분리 표기해 운영자 혼동을 줄였다.
- 삭제 액션을 감사 조회 allowlist와 테스트에 반영했다.

## 2. 원인 분석과 조치

### 원본 영상 수정 400

- 증상: 원본 영상 수정 요청이 `C0002` unknown 속성 오류로 실패했다.
- 원인: 프런트가 수정 불가 필드인 `bibleBookId`를 update payload에 포함했다.
- 조치: `SourceVideoUpdatePayload`에서 `bibleBookId`를 제거하고, 수정 요청에는 `title`, `videoUrl`, `durationSec`, `status`만 전송하도록 변경했다.

### 구간 2건만 저장되는 문제

- 증상: 실제 본문은 21절인데 구간 저장 후 2건만 보였다.
- 원인: 빈 구간 모달에 표시하던 기본 2건 JSON 예시를 그대로 저장했다.
- 조치: 빈 데이터는 `[]`로 열고, 예시는 placeholder로만 노출한다. 예시 JSON과 완전히 같은 내용은 저장하지 못하도록 가드를 추가했다.

### QT 클립 생성/조회 혼동

- 증상: QT 본문 ID와 클립 ID가 달라 운영자가 혼동할 가능성이 있었다.
- 확인: QT 본문 ID는 “어느 QT 본문에 붙는가”이고, 클립 ID는 `qt_video_clips` row의 고유번호라 일치할 필요가 없다.
- 조치: 목록 컬럼을 `클립 ID`, `QT 본문(날짜)`, `원본 영상 ID`로 바꿔 의미를 분리했다.

## 7. 남은 리스크와 PR 메모

- PR에는 로컬 DB의 `source_video #4`, `clip #4` 데이터가 포함되지 않는다. 리뷰어가 앱에서 확인하려면 같은 방식으로 원본 영상, 21개 구간, QT 클립을 생성해야 한다.
- 원본 영상 삭제는 물리 삭제이며 해당 원본의 절별 구간과 QT 클립도 함께 삭제된다. 운영자가 실수로 삭제하지 않도록 confirm 문구에 cascade 삭제 범위를 명시했다.
- MySQL auto increment는 삭제 후 1부터 되돌아가지 않는다. 클립 ID와 QT 본문 ID가 다를 수 있으므로 PR 본문에 이 점을 명시한다.
- 오늘 QT 화면은 `qtPassageId=2`인 경우 바로 영상 섹션을 볼 수 있다. 캐시가 남아 있으면 앱 새로고침 또는 service-bible 재시작 후 확인한다.
