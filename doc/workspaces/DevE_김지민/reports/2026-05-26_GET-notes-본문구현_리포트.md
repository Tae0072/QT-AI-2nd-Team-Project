## 2. 오늘 학습한 핵심 개념 12개

### 2-1. `@Query` JPQL — nullable 동적 쿼리 표준 관용구
`(:param IS NULL OR n.field = :param)` — 파라미터가 null이면 조건 우회, 값이 있으면 적용. 메서드 이름 폭발(2^N 메서드) 회피.

### 2-2. `@MappedSuperclass` — 공통 컬럼 묶음
DB 테이블 안 만들고, 자식 엔티티에게 컬럼 정의만 물려줌. BaseEntity가 그 패턴. → `deletedAt` 한 곳에 추가하면 모든 자식 테이블에 컬럼 생김.

### 2-3. `@SQLRestriction` — 자동 WHERE 추가
JPA의 모든 조회(`findById`/`findAll` 포함)에 자동으로 `WHERE deleted_at IS NULL` 추가. JpaRepository 기본 메서드도 자동 필터.

### 2-4. SRP — Service 레이어가 왜 필요한가
**Single Responsibility Principle**. Controller(HTTP) / Service(비즈니스+트랜잭션) / Repository(DB)로 책임 분리. 변경 비용·테스트 비용 ↓.

### 2-5. `SecurityContext` ↔ `@AuthenticationPrincipal`
- 필터가 `SecurityContextHolder.getContext().setAuthentication(...)` 으로 칠판에 적기
- Controller의 `@AuthenticationPrincipal`이 칠판에서 떼어 메서드 파라미터로 주입
- **호텔 비유**: 로비 직원(필터) → 안내 데스크 칠판(SecurityContext) → 청소부(Controller)에게 도우미(@AuthenticationPrincipal)가 전달

### 2-6. `@NotNull` vs `@NonNull`
- `@NotNull` (jakarta.validation): **런타임 검증**. `@Valid`와 함께. 위반 시 예외
- `@NonNull` (Spring lang): **정적 분석 메타데이터**. IDE·도구 경고만. 런타임 영향 X

### 2-7. SQL의 NULL 3법칙
`column = NULL` 절대 true 안 됨. `IS NULL` / `IS NOT NULL`만 가능. 동적 쿼리 `(:param IS NULL OR ...)`가 이걸 활용.

### 2-8. ArgumentCaptor vs `eq()`
- `eq()`: 호출 시점 즉시 비교 (단순)
- `ArgumentCaptor`: 캡처 후 자유 검증 (복잡한 객체·부분 검증·여러 호출)

### 2-9. 단위 vs 통합 테스트 — 격리 vs 진짜
- **단위**: Repository mock으로 막아두고 Service 로직만 검증
- **통합** (`@DataJpaTest`): 진짜 H2 DB + 진짜 JPQL 실행 → 단위가 못 잡는 JPQL 오타까지 검증

### 2-10. Soft Delete (소프트 삭제)
DB에서 실제로 안 지우고 `deleted_at` 시각만 찍음. **공유본 보존(F-10)**·**통계 정확성(F-13)**·**법적 보존(F-04)**에 필수.

### 2-11. JpaRepository 기본 메서드의 한계
`findById` 등 기본 메서드는 우리 JPQL과 무관하게 동작 → soft delete 필터 누락 가능. → `@SQLRestriction`으로 보강.

### 2-12. LIKE 와일드카드 이스케이프
사용자 입력의 `%`, `_`가 와일드카드로 해석되면 사고. JPQL `ESCAPE '\\'` 절 + Service 단 이스케이프 헬퍼.
