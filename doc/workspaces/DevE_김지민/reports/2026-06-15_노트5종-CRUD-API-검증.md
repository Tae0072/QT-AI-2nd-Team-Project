## 1. 배경 / 검증 동기

Flutter 앱을 로컬에서 직접 실행해 화면으로 확인하려 했으나 두 가지 장애가 있었다.

1. **Flutter SDK 버전 불일치** — 로컬 Flutter 3.41.7(Dart 3.11.5)인데 프로젝트 의존성 `flutter_quill 11.5.1`이 Dart 3.12.0+를 요구해 `pub get`/빌드가 막혔다. → Flutter 3.44.2로 업그레이드해 해결, 앱 빌드·기동까지는 성공.
2. **인증 토큰 부재** — 앱을 `--dart-define=DEV_FORCE_HOME=true`로 띄우면 로그인 단계를 건너뛰어 토큰이 없고, 인증이 필요한 화면(성경·QT·노트)이 `403 [M0002] 인증이 필요합니다`로 데이터를 못 받는다. (코드 결함 아님 — 개발용 단축 플래그의 부작용)

앱 UI 조작이 불가능한 상황이라, **노트 도메인 백엔드 API를 직접 호출**해 5개 카테고리의 작성·수정·저장·임시저장이 정상 동작하는지 전수 검증했다.

## 2. 검증 환경

| 항목 | 값 |
|---|---|
| 실행 형태 | 로컬 docker compose 스택(MSA), `SPRING_PROFILES_ACTIVE=docker` |
| 컨테이너 | gateway(8080), service-user(8081), service-bible(8082), service-note(8083), service-ai(8084), admin-server(8090), mysql, redis — 전부 healthy |
| 호출 경로 | 게이트웨이 `http://localhost:8080/api/v1/notes` |
| 인증 | service-user의 로컬 개발용 RS256 개인키(`JWT_PRIVATE_KEY`)로 USER Access Token 직접 발급(`sub=1`, `role=USER`, `type=access`). 회원은 `members` id=1(devadmin) |
| DB | 단일 `qtai` 스키마. `bible_verses` 31,104건 적재. `qt_passages`는 비어 있어 검증용 1건 시딩 |

> docker 프로파일은 dev-bypass가 꺼져 있어(정식 인증) 카카오 실로그인 없이 토큰을 얻을 수 없다. 로컬 키가 컨테이너 env에 노출돼 있어 USER 토큰을 직접 서명해 사용했다. 운영/공유 환경에는 적용 불가한 로컬 한정 테스트 기법이다.

## 3. 검증 시나리오

각 카테고리에 대해 동일한 3단계 흐름을 수행했다.

1. **생성 = 임시저장**: `POST /api/v1/notes` (`status=DRAFT`) → `201 Created`, `status=DRAFT` 기대
2. **수정 + 저장**: `PATCH /api/v1/notes/{id}` (본문 변경 + `status=SAVED`) → `200 OK`, `status=SAVED` 기대
3. **재조회 검증**: `GET /api/v1/notes/{id}` → 수정 본문 반영 + `status=SAVED` 기대

카테고리별 필수 입력(검증 코드 `NoteService.validateForSave` 기준):

| 카테고리 | enum | 필수 | 제약 |
|---|---|---|---|
| QT 노트 | `MEDITATION` | `qtPassageId` | qt readable 검증(`GET /qt/passages/{id}` 2xx) + 하루 1건(`active_unique_key='ACTIVE'`) |
| 설교 노트 | `SERMON` | `verseIds`(1개 이상) | `qtPassageId` 금지 |
| 기도제목 | `PRAYER` | 본문 | 자유(좌표 없음) |
| 회개 노트 | `REPENTANCE` | 본문 | 자유(좌표 없음) |
| 감사일기 | `GRATITUDE` | 본문 | 자유(좌표 없음) |

## 4. 검증 결과 — 전체 PASS (15/15)

| 카테고리 | ① 생성(임시저장 DRAFT) | ② 수정+저장(SAVED) | ③ 재조회 검증 |
|---|---|---|---|
| MEDITATION (QT 노트) | ✅ 201 / DRAFT | ✅ 200 / SAVED | ✅ 본문 수정 반영 |
| SERMON (설교 노트) | ✅ 201 / DRAFT | ✅ 200 / SAVED | ✅ 본문 수정 반영 |
| PRAYER (기도제목) | ✅ 201 / DRAFT | ✅ 200 / SAVED | ✅ 본문 수정 반영 |
| REPENTANCE (회개 노트) | ✅ 201 / DRAFT | ✅ 200 / SAVED | ✅ 본문 수정 반영 |
| GRATITUDE (감사일기) | ✅ 201 / DRAFT | ✅ 200 / SAVED | ✅ 본문 수정 반영 |

### 추가로 확인한 핵심 동작

- **DB 영속 확인**: API 응답 echo가 아니라 `notes` 테이블에 5건 모두 `status=SAVED` + `saved_at` 기록된 것을 직접 조회로 확인.
- **임시저장 ↔ 저장 양방향 전환**: `SAVED → DRAFT`(다시 임시저장)도 `200 OK` 정상. 자동 저장 없음(명시적 버튼 동작만, F-03 정책 부합).
- **설교 노트 절 갱신(@멘션 메타)**: `note_verses`가 `[1,2] → [1,2,3]`으로 정상 교체(수정 시 요청 `verseIds` 기준 재구성).
- **QT 노트 하루 1건 제한**: 같은 `qtPassageId`로 재생성 시 `409 N0002 이미 저장된 노트가 있습니다` 가드 정상. `active_unique_key='ACTIVE'` 부여 확인.
- **자유 노트 좌표 정책**: 기도·회개·감사는 `qtPassageId` 없이 저장되고 `active_unique_key=NULL`(중복 제한 없음) — 설계대로.

## 5. 결론

- **5개 카테고리(QT·설교·기도·회개·감사)의 작성·수정·저장·임시저장 백엔드 기능은 모두 정상이다.**
- 앱 화면에서 보였던 `403`은 노트 기능 결함이 아니라, `DEV_FORCE_HOME=true`로 로그인을 건너뛰어 **토큰이 없었던 것**이 원인이다. 유효한 USER 토큰이 있으면 위와 같이 전부 동작한다.
- 실제 앱에서 데이터가 채워진 노트 화면을 확인하려면 `DEV_FORCE_HOME` 없이 정상 카카오 로그인 흐름으로 실행해야 한다.

## 6. 검증 중 발견한 운영/도구 메모

- **Flutter 로컬 환경**: 팀 의존성(`flutter_quill 11.5.1`)이 Dart 3.12+를 요구하므로 로컬 Flutter는 **3.44.2 이상** 유지 필요. 3.41.x에서는 빌드 차단.
- **Git Bash + curl 한글 본문**: 인라인 `-d '{한글...}'`은 인코딩이 깨져 `C0002`(본문 파싱 실패). `printf '%s' '{...}' > file` 후 `--data-binary @file`로 보내야 한다.

## 7. 검증 위해 로컬 DB에 주입한 테스트 데이터 (정리 대상)

> 본 검증은 로컬 docker DB에만 영향을 주며 코드/스키마 변경은 없다.

| 테이블 | 주입 내용 | 비고 |
|---|---|---|
| `qt_passages` | id=1 (`테스트 QT 본문`, 오늘 날짜, status=ACTIVE) | MEDITATION readable 검증용. 유지하면 앱 QT 노트 화면 동작에 도움 |
| `qt_passage_verses` | qt_passage 1 ↔ verse 1,2 | 위 본문 연결 절 |
| `notes` / `note_verses` | 회원 devadmin 테스트 노트 5건(id 7~11) | 순수 테스트 데이터 — 삭제 권장 |

정리 SQL 예시:

```sql
-- 테스트 노트만 삭제 (qt_passage는 유지 권장)
DELETE FROM note_verses WHERE note_id BETWEEN 7 AND 11;
DELETE FROM notes WHERE id BETWEEN 7 AND 11;
```
