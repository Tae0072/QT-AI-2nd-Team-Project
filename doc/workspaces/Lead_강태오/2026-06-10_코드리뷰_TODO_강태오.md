## TODO 2. [P1] nginx 리버스 프록시 — 8080 진입점 복구

**문제**: flutter-app은 전 기능을 `:8080/api/v1`로 호출하는데(#465 모놀리식 제거 후) 8080에 듣는 서비스가 없음. admin-web 로그인도 8081 소관인데 프록시가 전부 8090으로 감. → 클라이언트 2종 모두 dev 토폴로지에 연결 불가.

**해결책**: docker-compose에 nginx 서비스 추가, 경로 prefix로 분배.

```nginx
# deploy/nginx/dev.conf (신규)
server {
  listen 8080;
  # 주의: 구체 경로를 먼저 — admin/auth는 service-user 소관
  location /api/v1/admin/auth/ { proxy_pass http://service-user:8081; }
  location /api/v1/admin/      { proxy_pass http://service-admin:8090; }
  location /api/v1/system/     { return 403; }  # 내부 전용 — 외부 진입 차단
  location ~ ^/api/v1/(auth|members|me|notifications|missions) { proxy_pass http://service-user:8081; }
  location ~ ^/api/v1/(bible|qt|study|music|praise)            { proxy_pass http://service-bible:8082; }
  location ~ ^/api/v1/(notes|sharing|reports|meditation)       { proxy_pass http://service-note:8083; }
  location /api/v1/ai/         { proxy_pass http://service-ai:8084; }
}
```

```yaml
# docker-compose.yml 추가
  gateway:
    image: nginx:1.27-alpine
    volumes: [ "./deploy/nginx/dev.conf:/etc/nginx/conf.d/default.conf:ro" ]
    ports: [ "${GATEWAY_PORT:-8080}:8080" ]
    depends_on: [ service-user, service-bible, service-note, service-ai, service-admin ]
    networks: [ qtai-net ]
```

- [ ] 경로→서비스 매핑을 실제 컨트롤러 `@RequestMapping` 전수와 대조해 확정(위 목록은 리뷰 기준 초안)
- [ ] `/api/v1/system/**` 외부 차단 포함(서비스 간 내부 호출은 컨테이너 네트워크 직통이라 영향 없음)
- [ ] `run-dev-web.ps1:50` "8080 대기"가 gateway 기준으로 자연 복구됨 — 주석만 갱신
- [ ] TTS: tts-service 코드가 저장소에 없음 — 운용 계획 결정 전까지 `app_config.dart`의 ttsBaseUrl(8090 충돌)은 이승욱과 8091 등으로 재지정 합의
- [ ] k8s에도 동일한 gateway(Deployment+Service) 추가 — 로컬 배포 한정 허용 범위(CLAUDE.md §8 예외) 내
- [ ] CLAUDE.md/문서에 "금지된 service-gateway 모듈 부활 아님 — 인프라 nginx"임을 명시(§8 Kubernetes 예외와 동일한 톤)

**검증**: compose 기동 후 `curl localhost:8080/api/v1/bible/books`(200), `curl -X POST localhost:8080/api/v1/admin/auth/kakao`(8081 도달 — 400/401이 아닌 카카오 토큰 검증 에러 응답), flutter 앱 dev 실행 스모크.
