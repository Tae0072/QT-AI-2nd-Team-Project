# 스터디 노트 — 서비스 간 "시스템 토큰" (입문자용)

> 2026-06-10. 배치/스케줄러가 다른 서비스를 부를 때 어떻게 인증하나.

## 1. 문제: 배치 호출엔 "사용자"가 없다
사용자 요청이면 그 사람의 JWT를 들고 다음 서비스로 전달하면 된다. 그런데 **매일 새벽 도는 배치/스케줄러**(예: AI 해설 생성 워커가 성경 본문을 읽음)는 "요청한 사용자"가 없다. 들고 갈 토큰이 없으니, 수신 서비스의 보호 경로를 통과할 수 없다. 그렇다고 인증을 비우면 보안 구멍이 된다.

## 2. 해결: "시스템 전용 토큰"을 따로 만든다
사용자 토큰과 **분리된** 시스템 전용 토큰을 둔다. 배치는 호출 직전에 이 토큰을 발급해 헤더에 실어 보낸다. 토큰 안에는 `sub=0`(사용자 아님), `role=SYSTEM_BATCH`를 담아 "나는 시스템 배치다"라고 표시한다. 수신 서비스는 이 토큰을 검증해 `SYSTEM_BATCH` 권한으로 통과시킨다.

## 3. 왜 사용자=RS256, 시스템=HS256인가 (핵심)
JWT 서명 방식 두 가지:
- **RS256(비대칭)**: 개인키로 서명, 공개키로 검증. **발급처가 하나**일 때 좋다 → 사용자 토큰은 `service-user`만 발급하니 RS256.
- **HS256(대칭)**: 하나의 **공유 시크릿**으로 서명·검증. **여러 곳이 서로 발급·검증**할 때 단순 → 시스템 토큰은 여러 서비스가 주고받으니 HS256 + 공유 시크릿.

둘을 **분리**하는 이유: 시스템 시크릿이 유출돼도 사용자 토큰(RS256)을 위조할 수 없다. 피해 범위를 가둔다. 게다가 시스템 토큰은 **단명(60초)**으로 만들어 탈취돼도 금방 만료되게 한다.

## 4. 필터의 "폴백" — 한 입구로 둘 다 받기
모든 서비스가 공유하는 `JwtAuthenticationFilter`는 이렇게 동작한다:
1. `Authorization: Bearer` 토큰을 **먼저 RS256(사용자)**로 검증 → 되면 사용자 인증.
2. 사용자 검증이 **실패하면** → **시스템 토큰(HS256)으로 폴백** 검증 → 되면 `SYSTEM_BATCH` 인증.
3. **둘 다 실패해야 401.**
포인트: 사용자 경로 동작은 1도 안 바뀐다. 시스템 폴백은 "추가"만 됐다. 그래서 기존 JWT 테스트가 안 깨진다(전 모듈 회귀 빌드로 확인).

## 5. "선택 주입"으로 기존 환경 안 깨기
시스템 토큰 검증기(`SystemTokenValidator`)는 `security.jwt.system-secret`이 설정된 서비스에만 빈으로 생긴다(`@ConditionalOnProperty`). 필터는 이 검증기를 **`ObjectProvider`로 선택 주입**해서, 시크릿이 없는 환경에선 검증기가 `null`이라 폴백을 그냥 건너뛴다. 즉 "있으면 쓰고 없으면 무시" — 점진 적용에 안전하다.

## 6. 시크릿은 절대 코드에 두지 않는다
HS256 공유 시크릿은 **env로만** 주입한다(`SECURITY_JWT_SYSTEM_SECRET`, Spring relaxed binding으로 `security.jwt.system-secret`에 매핑). 저장소·로그·예외 메시지에 시크릿이나 토큰 값을 남기지 않는다(CLAUDE.md §7·§9). 테스트에선 더미 시크릿(32바이트 이상 — HS256 최소 길이)을 쓴다.

## 7. SSoT 먼저 — 또 한 번의 거버넌스 교훈
지난번 k8s 가드처럼, 새 메커니즘도 **코드보다 CLAUDE.md(SSoT)부터** 갱신했다. "서비스 간 시스템 인증 = 공유 HS256 단명 SYSTEM_BATCH 토큰"을 §5에 명문화하고 결정 문서를 남긴 뒤 코드를 짠다. 안 그러면 리뷰가 "근거 문서 없음/SSoT 미갱신"으로 막는다.
