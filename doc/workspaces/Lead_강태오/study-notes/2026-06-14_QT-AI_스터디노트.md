# QT-AI 스터디 노트 — 이번 작업에서 배운 개념 (입문자용)

이 노트는 이번 작업에서 실제로 부딪힌 개념들을 **처음 보는 사람도 이해할 수 있게** 풀어 쓴 것이다.
"왜 이렇게 했는지"를 중심으로 정리한다.

---

## 1. Git 협업의 기본 규칙

- **브랜치(branch)**: 작업용 평행 우주. `dev`(공용)에서 새 가지를 쳐서 거기서 고치고, 다 되면 합친다.
- **직접 push 금지**: `dev`/`master`에 바로 밀지 않는다. 반드시 PR(합치기 요청)을 거친다.
- **브랜치 이름 규칙**: `종류/설명` 형식이고 종류는 정해진 단어만 쓴다(feature, bugfix, hotfix, chore, release, docs, test).
  - 실수 사례: `fix/...`로 만들었더니 CI가 "이름 규칙 위반"으로 막음 → `bugfix/...`로 다시 만들어 해결.
- **커밋 메시지 규칙(Conventional Commits)**: `feat:`(기능), `fix:`(버그), `docs:`(문서), `perf:`(성능) 등 접두어를 붙인다.
- **PR → CI → 머지**: PR을 올리면 자동 검사(CI)가 돈다. 전부 통과(녹색)해야 합친다.

## 2. CI가 자동으로 검사하는 것들

- **빌드/테스트**: 앱과 서버가 잘 컴파일되고 테스트가 통과하는지.
- **Requirements Guard**: 요구사항에서 금지한 것(예: AI 자유 챗봇)을 넣지 않았는지.
- **Gitleaks**: 비밀번호·토큰 같은 비밀이 코드에 새지 않았는지.
- **claude-review**: 자동 코드 리뷰.
- 핵심: "내 컴퓨터에선 됐는데?"를 막기 위해 **모두가 같은 자동 검사**를 통과해야 한다.

## 3. Flutter 앱에서 자주 쓴 개념

### 3-1. Riverpod (상태 관리)
- 화면이 보여줄 "데이터"와 그 데이터를 만드는 "로직"을 **Provider**라는 상자에 담아 둔다.
- `FutureProvider`: 서버에서 받아오는(시간이 걸리는) 데이터. 로딩/에러/완료 3상태를 자동으로 다룬다.
- 화면은 `ref.watch(provider)`로 그 상자를 구독하고, 값이 바뀌면 자동으로 다시 그린다.

### 3-2. 위젯 테스트와 "타이머 누수"
- 위젯 테스트는 화면을 가짜로 띄워 동작을 확인한다.
- 함정: 화면이 **주기적 타이머(Timer.periodic)**를 켜면, 테스트가 끝나도 타이머가 살아 있어 실패한다.
- 해결: 알림 폴러처럼 계속 도는 기능은 **실제 앱 실행(main 함수)에서만 켜지도록** 표시(`_launchedFromMain`).
  테스트는 main을 거치지 않으니 타이머가 안 켜진다.

### 3-3. 애니메이션 / 그림 직접 그리기(CustomPainter)
- 인트로(빅뱅→십자가→별)는 `AnimationController`로 0→1 진행값을 만들고,
  구간(Interval)별로 각 요소의 등장 타이밍을 나눈다.
- "빛이 한 점에서 퍼지고, 다시 T로 모이는" 느낌은 **RadialGradient(방사형 그라데이션)**의
  중심·반지름·투명도를 시간에 따라 바꿔서 표현했다. (단순한 원 확대가 아니라 가장자리가 번지게)

### 3-4. 동영상 플레이어
- `video_player`로 영상을 재생. 화면 어디든 한 번 탭하면 재생/정지가 토글되게 했다.
- 외부 영상 URL이 **403(접근 거부)**을 주면 재생이 안 된다 → 안정적으로 열리는 공개 영상으로 교체.

### 3-5. 기기 알림(로컬 노티)
- `flutter_local_notifications`로 상단 배너 알림을 띄운다. 첫 사용 시 OS 권한을 요청한다.
- 중요한 발견: 서버 알림을 기기 배너로 바꿔 주는 "폴러(poller)"가 **코드에는 있는데 켜는 곳이 없어서**
  한 번도 동작 안 했다. → 로그인 상태에서 폴러를 시작하고, 설정의 알림 토글과 연결했다.

## 4. 백엔드(MSA) 개념

### 4-1. MSA(마이크로서비스)와 도메인 경계
- 큰 서버를 기능별 작은 서버로 나눈다(회원/성경·QT/노트·나눔/AI 등).
- 규칙: 다른 도메인은 **공개 인터페이스(api/UseCase)로만** 부른다. 남의 내부 클래스를 직접 끌어다 쓰지 않는다.
  (그래야 한쪽을 고쳐도 다른 쪽이 안 깨진다)

### 4-2. 게이트웨이 라우팅(nginx)
- 앱은 항상 게이트웨이(8080) 한 곳만 부르고, 게이트웨이가 "주소의 앞부분(최장 접두사)"을 보고
  알맞은 서비스로 넘긴다.
- 함정: 새 주소 `/api/v1/me/*`를 추가했는데 게이트웨이 설정에 예외를 안 넣으면 **404**가 난다.
  (북마크/멘션 기능에서 실제로 겪고 예외 추가로 해결)

### 4-3. DB 마이그레이션(Flyway) vs 로컬 자동생성(ddl-auto)
- 운영은 **Flyway**로 스키마(테이블/컬럼)를 버전 관리한다.
- 로컬은 편의상 Hibernate가 자동으로 테이블을 만드는데(`ddl-auto=update`), 이때 byte[](이진 데이터)를
  **너무 작은 컬럼**으로 만들어 음성(mp3) 저장이 실패하는 함정이 있었다.
  → 엔티티에 `columnDefinition="LONGBLOB"`을 명시해 큰 데이터도 들어가게 했다.

### 4-4. 인증 토큰(JWT) 두 종류
- 사용자 토큰: **RS256**(공개키로 검증). 사용자 앱이 카카오로 로그인해 받는다.
- 시스템(배치) 토큰: **HS256**(공유 시크릿). 서버끼리의 작업에 쓰는 단명 토큰.
- 버그 사례: 로그인 요청에 **이전 만료 토큰**이 따라붙어 서버가 401로 거절 → 로그인 경로에는 토큰을 안 붙이게 수정.

### 4-5. "지연 캐시(lazy cache)"
- 비싼 작업(음성 생성)을 미리 다 하지 않고, **처음 요청될 때 한 번 만들어 DB에 저장**하고
  다음부터는 저장본을 바로 준다. 무료 호스팅의 느린 첫 응답(콜드스타트)을 그날 1회로 줄인다.

## 5. 디버깅(원인 찾기)의 사고법

- 증상 → **로그/DB/HTTP 상태코드**로 "어디까지 갔는지"를 본다.
  - 401/403: 주소·인증까지는 도달함. 404: 주소 매핑이 없음. 빈 화면: 데이터가 없음.
- 큰 교훈: **"기능이 안 된다"의 절반은 코드가 아니라 데이터/설정 문제**다.
  - QT영상 버튼이 꺼진 건 코드가 아니라 "승인된 영상 클립이 DB에 0건"이라서였다.
  - 알림이 안 온 건 폴러를 "켜는 코드"가 없어서였다.
