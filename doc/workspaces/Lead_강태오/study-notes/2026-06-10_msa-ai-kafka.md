# 스터디 노트 — AI 서비스 & Kafka 경계 (2026-06-10, service-ai)

> 입문자 기준. (작업하며 보강)

## 1. 왜 AI만 Kafka를 쓰나
대부분 서비스는 "조회"라 동기 RestClient면 충분하다. 그런데 AI 해설 생성은 **무겁고 비동기**(시간이 걸리고, 실패/재시도가 필요)다. 그래서 "할 일(job)"을 큐(Kafka)에 넣고 워커가 처리하는 패턴이 어울린다. 회의 결정도 "Kafka는 쓰기·잡 처리에만, 즉 AI에만"이다.

## 2. outbox → Kafka relay (왜 바로 안 보내나)
DB에 job을 저장하는 트랜잭션과 Kafka 전송이 따로 놀면, DB는 됐는데 Kafka 전송이 실패하는 불일치가 생긴다. 그래서 **DB의 outbox 테이블에 먼저 기록(같은 트랜잭션)** 하고, 별도 relay가 그걸 읽어 Kafka로 보낸다. 이러면 "전송 누락" 대신 "재처리 가능 상태"가 남는다.

## 3. AI에 절대 안 되는 것 (제품 규칙)
- 자유 챗봇 / 다중 턴 대화 / SSE / `/ai/sessions/**` — 임시 구현도 금지.
- RAG / vector DB — 금지.
- 허용은 **사전 생성/검증 + F-15 단발(single-turn) 사실 Q&A**(단어·시대상·역사배경)뿐.
- 외부 AI가 준 원문은 **검증을 통과하기 전엔 사용자에게 안 보낸다**. 승인 안 된 산출물도 노출 금지.
- API key는 env로만, 로그에 절대 안 남긴다.

## 4. 스케줄러는 테스트에서 끈다
시드/배치 스케줄러(@Scheduled, ApplicationRunner)는 부팅·테스트 때 멋대로 돌면 곤란하다. `@ConditionalOnProperty`로 테스트에서 비활성화해, 컨텍스트 로드 스모크가 깨지지 않게 한다(service-bible의 music 시드와 같은 방식).
