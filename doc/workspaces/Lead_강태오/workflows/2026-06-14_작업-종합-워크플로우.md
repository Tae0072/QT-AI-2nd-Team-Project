# QT-AI 작업 종합 워크플로우 (2026-06-14)

이 문서는 위 기간 작업이 **어떤 절차와 규칙으로 진행됐는지**를 정리한다. (무엇을 했는지는 종합 리포트 참고)

---

## 1. 저장소 / 브랜치 전략

- 문서 저장소(`2nd-Team-Project`)와 구현 저장소(`QT-AI-2nd-Team-Project`)를 분리 운용.
- 구현은 항상 `dev`에서 작업 브랜치를 따고, `dev`/`master`에 직접 push 금지.
- 브랜치명: `{type}/{scope}-{설명}` (type = feature|bugfix|hotfix|chore|release|docs|test).
  - 예: `feature/sharing-mentions`, `bugfix/tts-token-gate`, `docs/work-summary-2026-06-14`.
  - `fix/...`는 규칙 위반 → CI의 Branch Name Convention 실패(실제 경험: 한 번 `fix/`로 만들어 실패 → `bugfix/`로 재생성).

## 2. 작업 1건의 표준 흐름

```
요구 파악 → (필요 시) 선택지 질문 → 브랜치 생성(dev 기준)
→ 코드 변경 → flutter analyze / 관련 테스트 → 커밋(Conventional Commits)
→ push → PR(dev) → CI 통과 대기 → 머지(squash, 브랜치 삭제) → dev 동기화
→ 워크플로우/리포트 문서화
```

- 커밋 메시지: Conventional Commits(`feat`, `fix`, `perf`, `chore`, `docs`, `test`).
- PR은 가능하면 10파일/500라인 이하로 유지, 기능 PR은 관련 F-ID 명시.
- 검토는 "작업 후 2~3회" 원칙 — analyze + 테스트 + (필요 시) 로그/DB 직접 확인.

## 3. 검증 도구

- 앱: `flutter analyze <파일>`, `flutter test`(전체/범위), 위젯 테스트.
- 백엔드: `./gradlew -p qtai-server build/test`, ArchUnit/Modulith 경계 테스트.
- 운영 확인: `docker ps`, 컨테이너 로그, MySQL 직접 쿼리(원인 진단·데이터 확인).
- CI 게이트: Flutter/Server 빌드·테스트, Requirements Guard, Gitleaks, Spectral, claude-review.

## 4. 프런트(앱) 작업 패턴

- 상태관리는 Riverpod(Provider/FutureProvider/StateNotifier/AsyncNotifier).
- 화면 로직과 데이터 접근(repository) 분리, 위젯 단위 테스트로 회귀 방지.
- 테스트 안전: 주기 Timer/네트워크/플러그인은 **실제 main() 구동에서만** 활성화하여
  위젯 테스트(메서드 main 미경유)에서 타이머 누수·플러그인 LateError가 없도록 가드.
- 기기 OS 연동(알림 등)은 try/catch로 best-effort 처리해 앱 흐름을 막지 않음.

## 5. 백엔드(MSA) 작업 패턴

- 모듈러 모놀리식 + 서비스 분리(service-user/bible/note/ai, admin-server, 게이트웨이).
- 도메인 경계: 타 도메인은 `api/UseCase` 인터페이스로만 호출, Entity/Repo 직접 import 금지.
- 스키마는 admin-server(Flyway) 단독 소유. 새 컬럼/테이블은 마이그레이션으로 관리.
- 로컬은 Flyway가 사실상 off + Hibernate `ddl-auto=update`라, byte[] 컬럼이 작은 varbinary로
  생기는 함정이 있어 엔티티에 `columnDefinition="LONGBLOB"` 명시(경험: TTS 음성 저장 실패 수정).

## 6. 로컬 게이트웨이 / 라우팅 주의

- nginx 게이트웨이(8080)가 최장 접두사로 서비스 분배.
- 새 `/api/v1/me/*` 경로 추가 시 `deploy/nginx/dev.conf`에 예외를 넣지 않으면 404(C0004).
  (경험: `/me/bookmarks`, `/me/mentions`를 service-note로 라우팅하는 예외 추가)

## 7. 트러블슈팅에서 배운 패턴

- 증상 → 로그/DB/HTTP 상태코드로 원인 위치 특정 → 최소 변경 → 재검증.
  - 401/403 = 라우팅+인증까지 도달, 404 = 매핑/라우팅 누락, 403(외부 URL) = 자원 접근 차단.
- "기능이 안 된다"의 상당수는 코드가 아니라 **데이터 미시딩/설정 미연결**이 원인.
  - QT영상 버튼 비활성 = 승인 클립 0건, 알림 미수신 = 폴러가 시작 안 됨.

## 8. 문서화

- 작업 저장소 변경은 워크플로우 + 리포트를 항상 남김(`doc/workspaces/Lead_강태오/`).
- 리뷰성/진단성 작업은 채팅에서 정리·합의 후 필요 시 문서화(문서 자동 남발 금지).
