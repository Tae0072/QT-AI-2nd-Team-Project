## 2. AD-04 — 신고 처리(RESOLVE) / 반려(REJECT) + 감사로그

완료 기준: 처리·반려 결과가 반영되고, 감사 로그(`REPORT_RESOLVE`/`REPORT_REJECT`)와 신고자 알림(`REPORT_RESULT`)이 연동된다.

```powershell
# 2-1. 접수 목록에서 샘플 신고 id 확인
$rep = Invoke-RestMethod -Headers $H -Uri "$base/api/v1/admin/reports?status=RECEIVED&page=0&size=20"
$rep.data.content | Select-Object id, targetType, reason, status

# 2-2. POST 신고 → 처리 완료(RESOLVE), 신고자 알림 발송 on
$postReport = ($rep.data.content | Where-Object targetType -eq 'POST')[0].id
Invoke-RestMethod -Method Post -Headers $H -ContentType "application/json" `
  -Uri "$base/api/v1/admin/reports/$postReport/resolve" `
  -Body (@{ action = "HIDE_TARGET"; reason = "QA 처리 검증(대상 숨김 기록)"; notifyReporter = $true } | ConvertTo-Json)

# 2-3. COMMENT 신고 → 반려(REJECT)
$cmtReport = ($rep.data.content | Where-Object targetType -eq 'COMMENT')[0].id
Invoke-RestMethod -Method Post -Headers $H -ContentType "application/json" `
  -Uri "$base/api/v1/admin/reports/$cmtReport/reject" `
  -Body (@{ action = "NO_ACTION"; reason = "QA 반려 검증"; notifyReporter = $false } | ConvertTo-Json)
```

확인(SQL):

```sql
-- 상태 전이
SELECT id, target_type, status, processed_by_admin_id, processed_at FROM reports WHERE detail LIKE 'QA 샘플%';
-- 감사로그 연동
SELECT action_type, target_type, target_id, actor_label, created_at
FROM audit_logs WHERE action_type IN ('REPORT_RESOLVE','REPORT_REJECT') ORDER BY id DESC LIMIT 5;
-- 신고자 알림(REPORT_RESULT)
SELECT member_id, type, title, created_at FROM notifications WHERE type = 'REPORT_RESULT' ORDER BY id DESC LIMIT 5;
```

- [ ] 신고1 RESOLVED / 신고2 REJECTED
- [ ] audit_logs 에 REPORT_RESOLVE · REPORT_REJECT 2건
- [ ] notifications 에 REPORT_RESULT(신고자) 1건(처리 건만, 반려는 notifyReporter=false)

## 3. AD-06 — 시스템 공지 발행 → 사용자 알림 생성

완료 기준: 공지 발행 시 활성 사용자에게 알림(fanout)이 생성된다.

```powershell
# 3-1. 공지 등록(초안)
$notice = Invoke-RestMethod -Method Post -Headers $H -ContentType "application/json" `
  -Uri "$base/api/v1/admin/notices" `
  -Body (@{ title = "QA 점검 공지"; body = "AD-06 발행→알림 검증용 샘플 공지입니다."; status = "DRAFT" } | ConvertTo-Json)
$noticeId = $notice.data.id

# 3-2. 발행 → fanout
$pub = Invoke-RestMethod -Method Post -Headers $H -Uri "$base/api/v1/admin/notices/$noticeId/publish"
$pub.data   # 발행 결과(대상 수 등) 확인
```

확인(SQL):

```sql
SELECT member_id, type, title, link_type, created_at
FROM notifications WHERE type = 'NOTICE' ORDER BY id DESC LIMIT 10;
SELECT COUNT(*) AS notice_notifications FROM notifications WHERE type = 'NOTICE';
```

- [ ] 발행 응답의 대상 수 > 0 (활성 회원 수와 일치)
- [ ] notifications 에 type=NOTICE 행이 활성 회원 수만큼 생성
- [ ] 같은 공지 재발행해도 회원당 1건(중복 방지 `uk_notifications_member_event`)

## 4. AD-07 — 감사 로그 조회(추적성)

완료 기준: 위 2·3에서 만든 관리자 액션이 감사 조회 화면/목록에 노출·필터된다.

```powershell
# 신고 처리 액션 필터
```

- 관리자 웹 화면: 좌측 메뉴 **감사 로그(AD-07)** → 액션 유형 필터로 `REPORT_RESOLVE`, `REPORT_REJECT`, 공지 발행 액션 확인.
- [ ] 신고 처리/반려 액션이 목록·필터·상세에 노출

## 5. AD-10 — 배치 실행 로그(성공/실패 추적성)

완료 기준: 배치 성공/실패 로그가 목록·필터로 보이고, 실패행에서 실패시각·오류유형·재시도 판단이 가능하다.

```powershell
# 실패만 필터
$batch = Invoke-RestMethod -Headers $H -Uri "$base/api/v1/admin/ai/batch-run-logs?status=FAILED&page=0&size=20"
$batch.data.content | Select-Object batchName, status, failedCount, errorType, finishedAt
```

- 관리자 웹 화면: 좌측 메뉴 **배치 로그(AD-10)** → 상태 필터 `FAILED`/`SUCCEEDED`.
- [ ] SUCCEEDED · FAILED 샘플 2건 노출
- [ ] 실패행에 error_type(DeepSeekTimeoutException) · finished_at(실패시각) 표시 → 재시도 판단 가능
- [ ] "보류"였던 AD-10 추적성 판정 → **가능**으로 확정
