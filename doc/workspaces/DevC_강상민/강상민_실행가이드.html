0. 처음 켜면 여기부터

내가 지금 어디에 있는지 모르면 아래 순서만 따라간다.

1단계: 저장소 위치 확인

cd /d C:\workspace\QT-AI-2nd-Team-Project
git status --short --branch
  • ## dev...origin/dev이면 시작 가능하다.
  • 다른 브랜치면 먼저 dev로 이동한다.

2단계: 오늘 할 작업 패널 열기

  • AI 생성/검증 로그 모델 정리면 W1-1
  • AI UseCase 설계면 W1-2
  • 해설 검증 흐름 검토면 W1-3
  • 실패·재처리 기준 정리면 W1-4

3단계: workflow 먼저 작성

작업 기록 위치는 별도 구현 GitHub 기준으로 여기다.

workspaces/강상민/workflows/YYYY-MM-DD_task-name.md
workspaces/강상민/reports/YYYY-MM-DD_task-name_report.md
중요: 이 문서 저장소에는 실제 Spring Boot/Flutter/OpenAPI 구현 골격을 만들지 않는다. 실제 작업 기록은 별도 구현 저장소의 workspaces/강상민 기준으로 PR에 적는다.

1. 실제 저장소 지도

강상민이 자주 만지는 경로만 먼저 외우면 된다.

경로무엇을 하는 곳인가강상민이 할 일
qtai-server/src/main/java/com/qtai/domain/aiAI 도메인. api/internal/client/web 구조.해설 생성, 검증 로그, F-15 Q&A 흐름 보강.
qtai-server/src/main/java/com/qtai/external/llm외부 LLM 호출 공통 영역.DeepSeek 호출은 공유 external 경계로 분리.
qtai-server/src/test/java/com/qtai/domain/aiAI 도메인 테스트.생성/검증/실패 재처리 테스트 추가.

.github/workflowsGitHub Actions와 자동 리뷰.실패하면 로그를 보고 report에 남긴다.

2. 소유권

내가 마음대로 고칠 수 있는 곳과 협업 확인이 필요한 곳을 구분한다.

OWNER 강상민 주 담당 경로이 경로는 내가 작업 주도. 그래도 CODEOWNERS에 포함된 협업자는 PR에서 같이 본다.
경로CODEOWNERS 기준작업 예시
domain/ai강상민, 강태오해설 생성, 검증, F-15 Q&A
external/llm강상민, 강태오DeepSeek 등 외부 LLM 호출
domain/admin/ai강상민, 김지민, 강태오관리자 재생성·승인 화면 연동
domain/study김태혁, 강상민시뮬레이터 산출물 경계 협업
CHECK 다른 사람 확인이 필요한 순간내가 고쳐도 되나? 싶으면 여기 기준으로 판단한다.

이지윤 확인

  • 성경 본문 범위와 출처 메타데이터가 해설 입력에 닿을 때
  • 금지 번역본·검증 참조 원문 정책이 애매할 때

김태혁 확인

  • 시뮬레이터 산출물 생성·상태 모델과 AI 생성물이 닿을 때
  • READY/FAILED 재처리 기준이 연결될 때

김지민 확인

  • 관리자 웹 화면에서 AI 산출물 승인·재생성을 보여줄 때
  • 사용자 화면에 AI 기능처럼 보이는 문구가 생길 때

강태오 확인

  • 관리자 권한, 감사 로그, API 계약, 품질 게이트가 바뀔 때
  • 외부 LLM 설정·시크릿 경계가 닿을 때

3. 브랜치 생성 방법과 규칙

규칙: 항상 dev에서 새 브랜치를 만든다. master나 dev에 직접 커밋하지 않는다. 브랜치 이름은 {type}/{scope}-{short-description} 형식이다.

처음 브랜치 만들기

cd /d C:\workspace\QT-AI-2nd-Team-Project
git fetch origin
git checkout dev
git pull origin dev
git checkout -b feature/ai-run-log-and-admin-trigger

브랜치 이름 예시

  • feature/ai-explanation-batch — 해설 배치 생성
  • feature/ai-validation-log — 생성·검증 로그
  • feature/ai-f15-qa-validation — F-15 단발 Q&A 검증
  • feature/ai-admin-trigger — 관리자 재생성 트리거
  • test/ai-failure-retry — 실패·재처리 테스트
  • fix/ai-user-session-block — 사용자 세션형 AI 차단
주의 브랜치 만들 때 흔한 실수입문자가 가장 자주 막히는 지점

4. PR까지 가는 전체 순서

PR 8단계 PR 루틴작업마다 이 순서대로 반복한다.

1~4단계: 작업 전

  1. dev 최신화
  2. 브랜치 생성
  3. workspaces/강상민/workflows에 workflow 작성
  4. 작업 패널의 기준 문서 확인

5~8단계: 작업 후

  1. 구현 또는 문서/계약 수정
  2. 검증 명령 실행
  3. workspaces/강상민/reports에 report 작성
  4. push 후 PR 생성
git status --short
git add qtai-server/src/main/java/com/qtai/domain/ai workspaces/강상민
git commit -m "feat(ai): add generation validation log flow"
git push -u origin feature/ai-run-log-and-admin-trigger
gh pr create --base dev --head feature/ai-run-log-and-admin-trigger --title "feat(ai): add generation validation log flow"
CI 자동 검사와 실패 시 행동CI/Claude 리뷰가 실패하면 report에 남기고 고친다.
검사무엇을 보는가실패하면
spring-build서버 빌드/테스트로컬에서 ./gradlew -p qtai-server build
requirements-guard금지 기능/금지 데이터금지 단어와 기능 제거
gitleaks시크릿 노출토큰/비밀번호/키 제거
spectralOpenAPI 문법yaml 계약 수정
claude-pr-review자동 코드 리뷰REQUEST_CHANGES 사유를 반영
