# QT-AI 문서·명세 기준 저장소

QT-AI는 QT와 성경 묵상에 익숙하지 않은 사용자도 오늘의 본문을 쉽게 읽고, 한/영 성경과 사전 생성된 해설, 장면 시뮬레이터, 개인 묵상 기록을 한 흐름 안에서 사용할 수 있도록 돕는 QT 보조 앱입니다.

이 저장소는 실제 구현 코드 저장소가 아니라 **프로젝트 문서와 명세의 기준 저장소**입니다. Spring Boot, Flutter, DB, OpenAPI, CI/CD 같은 실제 구현 골격은 별도 구현 GitHub에서 생성하고, 이 저장소의 문서를 기준으로 맞춥니다.

## 백엔드 로컬 실행 (Docker Compose, MSA)

`qtai-server`는 단일 모놀리식이 아니라 **Spring 서비스 5개 + nginx 게이트웨이** 구조입니다. MySQL·Redis까지 한 번에 `docker compose`로 띄웁니다.

**구성과 포트**

| 컨테이너 | 호스트 포트 | 역할 |
| --- | --- | --- |
| gateway (nginx) | `8080` | 클라이언트 단일 진입점, 경로별로 각 서비스에 분배 |
| service-user | `8081` | 사용자/인증(member, notification, mission), JWT 발급 |
| service-bible | `8082` | 읽기전용 콘텐츠(bible, qt, study, music, praise) |
| service-note | `8083` | 노트/나눔/신고(note, sharing, report 제출) |
| service-ai | `8084` | AI(사전생성·검증, F-15 Q&A) |
| admin-server | `8090` | 관리자(모놀리식 복사 후 admin 컨트롤러만, 단일 DB 공유) |
| mysql | `3307` | DB(컨테이너 내부는 3306) |
| redis | `6380` | 캐시/토큰(컨테이너 내부는 6379) |

> Flutter 앱·admin-web 등 클라이언트는 게이트웨이 `8080` 하나만 호출합니다. 호스트 포트는 `.env`(`MYSQL_HOST_PORT`, `GATEWAY_PORT` 등)로 바꿀 수 있습니다.

**사전 준비**

- Docker / Docker Compose, **JDK 21**(jar 빌드용)
- 포트 `8080`, `8081~8084`, `8090`, `3307`(MySQL), `6380`(Redis)이 비어 있어야 합니다. 이미 사용 중이면 `.env`에서 해당 호스트 포트를 바꿉니다.

**실행 순서**

1. 비밀값을 담을 `.env`를 만듭니다. `.env`는 보안상 커밋되지 않습니다(`.gitignore` 등록).

   ```bash
   cp .env.example .env
   ```

   `.env`의 `JWT_PRIVATE_KEY` / `JWT_PUBLIC_KEY`(RS256, base64 DER)와 `SECURITY_JWT_SYSTEM_SECRET`을 채웁니다. Git Bash에서 `bash run-dev-web.sh`를 한 번 실행하면 dev용 throwaway 키가 `.env`에 자동 생성됩니다.

2. **각 서비스 bootJar를 먼저 빌드합니다.** 각 `Dockerfile`은 빌드된 jar를 복사만 하므로(컨테이너 안에서 컴파일하지 않음) 이 단계를 건너뛰면 이미지 빌드가 실패합니다.

   ```bash
   cd qtai-server
   ./gradlew bootJar      # service-user/bible/note/ai + admin-server 일괄 빌드
   cd ..
   ```

3. 전체 스택을 띄웁니다.

   ```bash
   docker compose up --build
   ```

   - 서비스는 MySQL이 준비(healthy)된 뒤 `docker` 프로파일로 기동하며, Flyway(admin-server 단독 소유)가 스키마를 생성합니다.
   - 접속(클라이언트 진입점): `http://localhost:8080`

4. 인프라(MySQL·Redis)만 띄우려면: `docker compose up -d mysql redis`

**원클릭 실행 스크립트(서버 + Flutter 웹)**

```bash
bash run-dev-web.sh          # Git Bash 권장: .env 자동 생성 → 인프라 → 서버 → Flutter 웹(3000)
# 또는 PowerShell
powershell -ExecutionPolicy Bypass -File .\run-dev-web.ps1
```

**프로파일별 DB**

| 프로파일 | DB | 용도 |
| --- | --- | --- |
| `test` / `local` | H2 (인메모리) | 단위·통합 테스트, 인프라 없이 빠른 개발 |
| `docker` / `dev` | MySQL (Docker) | 실DB 확인·시연 |
| `prod` | MySQL (`DB_URL` 환경변수 주입) | 운영 배포(주소만 바꾸면 클라우드 전환 가능) |

> 테스트만 돌릴 때는 Docker 없이 `./gradlew -p qtai-server test`로 충분합니다(H2 사용).

> 단일 서비스만 개발할 때는 인프라를 띄운 뒤 `cd qtai-server && SPRING_PROFILES_ACTIVE=dev ./gradlew :service-user:bootRun` 처럼 모듈을 지정해 실행합니다.

## 프로젝트 한 줄 정의

QT를 처음 접하는 사용자도 오늘의 본문을 바로 읽고, 쉬운 해설과 장면 시뮬레이터를 참고하며, 자기 묵상 기록을 남기고 필요할 때만 닉네임으로 나눌 수 있는 심플한 큐티 앱.

## 왜 만드는가

| 문제 | QT-AI의 해결 방식 |
| --- | --- |
| QT를 처음 시작하는 사용자는 어디서부터 읽어야 할지 막막하다 | 앱 첫 진입을 오늘의 QT 화면으로 두고, 본문 → 해설 → 시뮬레이터 → 묵상 기록 순서로 안내한다 |
| 성경 본문만으로는 배경과 의미를 이해하기 어렵다 | 한/영 본문, 쉬운 요약, 용어 풀이, AI가 사전 생성한 해설을 선택적으로 제공한다 |
| AI 챗봇 답변은 출처와 검증이 불안정할 수 있다 | 자유 챗봇은 제공하지 않고, 사전 생성·검증한 해설과 단발·검증형 사실 기반 Q&A만 노출한다 |
| 묵상 앱이 커뮤니티 피드 중심이 되면 개인 기록이 흐려진다 | 기본은 비공개 개인 묵상이며, 사용자가 선택할 때만 닉네임 나눔으로 전환한다 |
| 성경·주석·찬양 데이터는 저작권 리스크가 크다 | 출처·라이선스·재배포 가능 여부를 확인한 데이터만 쓰고, 가사·음원·금지 번역본은 저장하지 않는다 |

## MVP 사용자 흐름

1. 사용자는 앱을 열면 별도 홈 화면 없이 오늘의 QT 본문으로 진입한다.
2. 오늘 본문을 한글/영어 절 단위로 읽고, 필요하면 쉬운 해설과 용어 풀이를 연다.
3. 시뮬레이터 상태가 `READY`이면 장면 보기 버튼을 눌러 본문 장면을 팝업으로 확인한다.
4. 느낀 점, 기억할 구절, 적용할 점, 기도 제목을 묵상 노트에 저장한다.
5. 마이페이지에서 묵상 달력과 기록 리스트로 내 묵상 흐름을 확인한다.
6. 원할 때만 묵상 일부를 닉네임 나눔 공간에 공개하고, 찬양은 운영자 큐레이션 목록에서 내 목록에 저장한다.

| F-02 | AI 콘텐츠 사전 생성 | 04:00 KST 배치 또는 관리자 트리거로 해설·시뮬레이터를 미리 생성 |
| F-03 | 묵상 노트 | 오늘 QT 기준으로 개인 묵상을 생성, 수정, 삭제, 조회 |
| F-04 | 회원 인증 | Kakao 로그인과 JWT 기반 자동 로그인 제공 |

| F-06 | 관리자 운영 | 오늘 QT, 승인 해설, 시뮬레이터 상태, 신고, 찬양 큐레이션 관리 |
| F-07 | 첫 실행 튜토리얼 | QT, 해설, 묵상, 적용, 기도 제목 등 기본 용어 안내 |
| F-08 | 쉬운 해설 | 사용자에게 노출 가능한 요약·해설·용어 풀이 제공 |
| F-09 | 찬양 큐레이션 | AI 추천 없이 운영자가 등록한 곡 목록에서 사용자가 저장 |
| F-10 | 닉네임 나눔 | 사용자가 선택한 묵상만 닉네임으로 공유, 좋아요·댓글·신고 최소 제공 |
| F-12 | 시뮬레이터 보기 | 오늘 QT 본문 옆 상태 기반 버튼 제공, `READY`만 활성화 |
| F-13 | 묵상 달력 | 마이페이지에서 월별 묵상 여부와 날짜별 기록 이동 제공 |

## 고정된 제품 기준

- 요구사항 충돌 시 [07_요구사항_정의서.md](./07_요구사항_정의서.md) v3.1을 우선합니다.
- QT 본문 공개 시각은 성서 유니온 기준 00:00 KST입니다.
- 우리 시스템의 QT 범위 수집 배치 시각은 04:00 KST입니다.
- 00:00~04:00 KST 구간에는 새 범위와 AI 산출물이 준비되지 않았을 수 있으므로 기존 캐시를 유지합니다.
- Today QT 100%는 본문, 해설 진입점, 묵상 진입점, 시뮬레이터 상태값이 정상 응답한다는 뜻입니다.
- 모든 본문에 시뮬레이터 클립이 반드시 존재해야 한다는 의미는 아닙니다.
- 시뮬레이터 상태가 `READY`일 때만 보기 버튼을 활성화합니다.
- `MISSING`, `FAILED`, `DISABLED` 상태에서는 시뮬레이션 보기 버튼을 비활성화합니다.
- 저작권 관련 표현은 "저작권 리스크를 낮춘다"로 통일합니다.
- 이벤트 실패 처리는 v1에서 "핸들러 실패 로그 + 재처리 가능" 수준으로 현실화합니다.

## AI 사용 원칙

QT-AI의 AI는 사용자와 실시간으로 대화하는 챗봇이 아닙니다.

| 구분 | 기준 |
| --- | --- |
| 허용 | 오늘 QT 승인 해설 생성, 시뮬레이터 클립 생성, 관리자 재생성 트리거 |
| 실행 시점 | 매일 04:00 KST QT 범위 수집 이후 배치 또는 관리자 수동 트리거 |
| 사용자 노출 | 검증된 해설과 시뮬레이터 산출물 조회만 허용 |
| 금지 | AI 자유 챗봇, 다중 턴 대화, SSE, `/ai/sessions/**`. 해설·시뮬레이터는 사용자 요청 시 생성 금지 |

해설 데이터는 `commentary_sources`, `commentary_materials`, `commentary_material_verses`, `verse_explanations`, `glossary_terms`, `validation_reference_jobs` 기준으로 관리합니다. 한국어 참조 원문은 검증용으로 보호하고, 사용자에게는 검증 완료된 `verse_explanations`만 노출해 저작권 리스크를 낮춥니다.

## 기술 방향

| 영역 | v1 확정 기준 | v1에서 하지 않는 것 |
| --- | --- | --- |
| 백엔드 | Java 21, Spring Boot 3.3, `service-user/bible/note/ai` + `admin-server` 5개 서비스 + nginx 게이트웨이(MSA, 단일 DB 공유) | 서비스별 DB 분리 |
| 도메인 통신 | 도메인 간 Entity/Service 직접 import 금지, DTO/Interface 통신 | 도메인 내부 구현 직접 참조 |
| DB | MySQL 8.0 | PostgreSQL |
| 검색 | RDB 인덱스 | RAG, ChromaDB, vector DB, Elasticsearch |
| 이벤트 | Spring `ApplicationEventPublisher`(서비스 간은 RestClient) | Kafka(AI 영역 외 확장) |
| 인프라 | Docker Compose, 로컬 한정 Kubernetes/Helm 매니페스트(`k8s/`) | 운영(프로덕션) K8s/Helm 도입 |
| LLM | DeepSeek API, 배치·관리자 트리거 전용 | 사용자 요청 경로 LLM 호출 |
| 앱 | Flutter Android/디버그 빌드 시연 | iOS 정식 배포 |

## 구현 저장소 반영 원칙

이 저장소는 기준 틀입니다. 실제 구현 저장소에서는 다음 원칙을 지켜야 합니다.

1. `07_요구사항_정의서.md` v3.1과 충돌하는 구현을 만들지 않습니다.
2. 외부 공개 API는 `/api/v1/**`로 통일하고, 내부 도메인 호출은 Java Interface로 분리합니다.
3. 도메인 간 직접 import 금지 규칙을 PR 검증에 연결합니다.
4. 해설·시뮬레이터 AI는 배치·관리자 트리거로 구현하고, F-15 사실 기반 Q&A는 단발·검증 흐름으로 구현합니다.
5. 성경·주석·찬양 데이터는 라이선스와 출처 표기 기준을 통과한 뒤 적재합니다.
6. 문서와 구현이 달라지면 구현을 우선 고치는 것이 아니라, 기준 문서 변경 필요성을 먼저 검토합니다.
